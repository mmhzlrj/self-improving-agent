## DeepSeek Web Provider 移植进展

### 尝试的方法

1. **直接在 18789 配置 deepseek-web** - 失败，因为 18789 不支持 `deepseek-web` API 类型

2. **用 Channel 方式接入** - 讨论中，发现 Channel 是通信渠道，不是模型 provider

3. **创建 Provider 扩展** - 开始创建 `~/.openclaw/extensions/deepseek-web/`，参考 minimax-portal-auth 的方式

### 核心问题

- 原来的 18789 Gateway 只支持固定的 API 类型（openai-completions, anthropic-messages 等）
- deepseek-web 需要自定义认证（Cookie + Bearer Token + PoW 挑战）
- zero-token 是修改了核心 agents 代码来支持的，不是作为插件

## DeepSeek Web 认证移植到 18789 Gateway

### 背景
用户希望在 18789 Gateway 上复现 zero-token 的功能，数据隐私是最高优先级。

### 步骤与结果

#### 1. 克隆 zero-token 源码 ✅
- 位置：`~/.openclaw/workspace/openclaw-zero-token/`

#### 2. 创建扩展目录 ✅
- 位置：`~/.openclaw/extensions/deepseek-web/`

#### 3. 复制核心代码 ✅
- `deepseek-web-auth.ts` - 认证逻辑
- `deepseek-web-client.ts` - 完整客户端（含 PoW 解决方案）

#### 4. 测试 Chrome 18800 连接 ✅
- 成功连接到 Chrome 18800 端口
- 获取 DeepSeek Cookie ✅
- 获取 Bearer Token ✅
- 凭证保存到 `deepseek-credentials.json`

#### 5. API 调用测试
- 创建 Chat Session ✅
- 发送消息 ❌ (40300 MISSING_HEADER 错误)

#### 6. 调试 40300 错误
- 尝试添加更多 HTTP Headers
- 正确解析 Bearer Token (从 localStorage)
- 发现 Bearer Token 需要从**网络请求拦截**获取（而非 localStorage）
- 发现 zero-token 使用的是 CDP Port 9222（而非 18800）
- 错误原因：可能是 DeepSeek API 最近更新

### 关键代码文件

| 文件 | 功能 |
|------|------|
| `~/.openclaw/extensions/deepseek-web/auth-get.mjs` | 获取凭证脚本 |
| `~/.openclaw/extensions/deepseek-web/test-correct.mjs` | API 测试脚本 |
| `~/.openclaw/extensions/deepseek-web/deepseek-credentials.json` | 保存的凭证 |
| `~/.openclaw/extensions/deepseek-web/deepseek-web-client.ts` | 完整客户端（含 PoW）|
| `~/.openclaw/extensions/deepseek-web/deepseek-web-auth.ts` | 认证逻辑 |

### 技术细节

**认证流程**：
1. 连接到 Chrome CDP (18800)
2. 获取 chat.deepseek.com 的 cookies
3. 从 localStorage 或网络请求获取 Bearer Token
4. 解析 Bearer Token（需要提取 value 字段）

**PoW 解决方案**：
- 已移植完整代码到 `deepseek-web-client.ts`
- 包含 SHA3 WASM 模块（约 300 行 Base64 编码）
- 支持 CPU 计算作为备选

### 待解决
- 40300 MISSING_HEADER 错误 - DeepSeek API 可能最近更新
- 可能需要使用 zero-token 的 onboard 流程获取正确凭证
- 或等待 zero-token 项目更新
