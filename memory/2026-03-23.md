## 深度调研 zero-token 项目（13:00）

### 关键发现
1. **SSE 完成检测**：`reader.read()` 循环直到 `done: true`，不是用时间限制
2. **Token 来源**：Kimi 从 localStorage 读 `access_token`（不是 `kimi-auth` cookie）
3. **API 调用位置**：zero-token 所有 API 调用都在 `page.evaluate()` 里，浏览器自动带 cookie
4. **我们的根本问题**：用 Node.js fetch，cookie 不会自动附加

### 修复计划
1. Kimi：改用 localStorage 的 `access_token`，不用 kimi-auth cookie
2. GLM：加 token refresh 逻辑，读 SSE 直到 done
3. Qwen：改用 page.evaluate() + Bearer token，修复 API URL/格式
4. 所有平台：API 调用移入 page.evaluate()

## Doubao MCP Server 凭证获取方案（最终成功）

### 问题
- Cookie 存在 Chrome 加密的 SQLite 里，Node.js 无法直接读
- `chromium.launchPersistentContext` 在 macOS 上会挂起（`browser.close()` 无法返回）
- Node.js Playwright 连接 Chrome 需要 remote debugging 端口

### 解决方案
1. 用 Python Playwright（不是 Node.js）通过 CDP 连接 Chrome
2. Chrome 用"用户 2"Profile 启动：`--remote-debugging-port=9222 --user-data-dir=".../Chrome/Profile 2"`
3. 从 `sessionid` Cookie 获取凭证
4. 保存到 `creds.json`

### 关键命令
```bash
# 启动 Chrome 用户2
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222 \
  --user-data-dir="$HOME/Library/Application Support/Google/Chrome/Profile 2"

# Python Playwright 连接
browser = p.chromium.connect_over_cdp("http://localhost:9222")
```

### 经验教训
- **用 Python Playwright 而不是 Node.js** - `browser.close()` 在 Node.js 里会挂起，但 Python 的 `sync_playwright` 可以正常关闭
- **Chrome Profile 2 是已登录豆包的 Profile**
- **sessionid 就是 Refresh Token**，有效期约 7-30 天，过期后重新运行脚本获取即可
