## 下午 15:53-16:03 进展

### ✅ Chrome 18800 调试端口终于打开

**方法**（macOS Chrome Singleton 问题解决）：
```bash
# 1. 先杀掉所有 Chrome
killall "Google Chrome" 2>/dev/null

# 2. 删掉 Singleton 锁文件
rm -f ~/Library/Application\ Support/Google/Chrome/SingletonLock
rm -f ~/Library/Application\ Support/Google/Chrome/SingletonCookie
rm -f ~/Library/Application\ Support/Google/Chrome/SingletonSocket

# 3. 用独立 user-data-dir 启动（避免跟主 Chrome 冲突）
nohup "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" \
  --user-data-dir=/tmp/chrome-debug-18800 \
  --remote-debugging-port=18800 \
  --disable-gpu \
  --no-sandbox \
  --headless \
  > /tmp/chrome-18800.log 2>&1 &

# 4. 验证
curl http://127.0.0.1:18800/json/version
```

**教训**：
- 不能用 `--profile-directory` 加默认 user-data-dir（会冲突）
- 必须用独立 user-data-dir
- Chrome 在跑时Singleton锁会阻止新实例，必须删锁文件

- 响应: SSE 格式，`event_data` 内嵌双层 JSON

**GLM X-Sign 方案：**
- X-Sign 算法：`md5(timestamp + '-' + nonce + '-' + SIGN_SECRET)`
- SIGN_SECRET: `8a1317a7468aa3ad86e997d08f3f31cb`
- Timestamp 算法：修改时间戳末位（各位和 - 倒数第二位 % 10）
- API: `https://chatglm.cn/chatglm/backend-api/assistant/stream`
- 需先 refresh token 获取 access_token

## 23:00-23:06 千问 WebAuth 最终完成

### 四平台全部完成 ✅

| Platform | Tool | 协议 | 认证方式 | 响应格式 |
|---|---|---|---|---|
| Kimi | kimi_chat | Connect RPC | Bearer Token (kimi-auth JWT) | `block.text.content` |

| GLM | glm_chat | SSE | X-Sign MD5 + Bearer Token | `parts[].content[].text` |
| **Qwen** | qwen_chat | SSE | **override window.fetch 拦截** | **`choices[0].delta.content`** |

### Qwen 关键突破

**问题**：千问的 Authorization header 是页面 JS 用内部变量加的，Playwright 所有层面（route/CDP Network/page.on request）都看不到。

**方案**（参考 zero-token 思路）：
1. Override `window.fetch` 在流级别拦截 SSE 响应
2. 页面 JS 发请求时自动带 Cookie 和内部 Token（外部无法注入）
3. 解析 SSE 格式：`choices[0].delta.content`
4. 拼接所有增量片段得到完整文本

**代码模式**：
```javascript
window._qwenText = '';
window._qwenSSE = null;
const _origFetch = window.fetch;
window.fetch = async (...args) => {
  const response = await _origFetch(...args);
  const newStream = new ReadableStream({
    async start(controller) {
      while (true) {
        const { done, value } = await reader.read();
        if (done) { controller.close(); break; }
        const chunk = decoder.decode(value, { stream: true });
        // 解析 SSE，提取 choices[0].delta.content
        controller.enqueue(value); // 同时传回给页面
      }
    }
  });
  return new Response(newStream, { headers: response.headers, status: response.status });
};
```

### 踩的坑

1. **Qwen API URL 错误**：最初以为用 dashscope，实际是 `chat.qwen.ai/api/v2/chat/completions`
2. **parent_id 状态管理**：最初想复用已有 chat_id，但每次新建 chat 更简单（zero-token 方式）
3. **Authorization 无法捕获**：页面 JS 在 fetch 内部加 header，Playwright 的 network interception 看不到内部 headers
4. **SSE 格式错误**：最初以为用 `text` 字段，实际是 `choices[0].delta.content`
5. **重复文本**：SSE 响应有 incremental_output=true，同一个 text 会多次发送，去重逻辑要判断增量

### 错误记录

| 错误 | 原因 | 解决 |
|---|---|---|
| 千问 401 Unauthorized | qwen_session 是 HttpOnly，无法从 document.cookie 获取 | 用 override window.fetch 方案 |
| SSE 解析 text 为空 | Qwen 用 `choices[0].delta.content` 而非 `text` | 修正解析路径 |
| Authorization always undefined | 页面 JS 在 fetch 内部加 header，route 拦截看不到 | 改用 window.fetch override |
| 重复累积文本 | incremental_output=true，同一内容多次推送 | 保留所有增量片段拼接 |
| goto timeout | chat.qwen.ai 有持续网络活动，waitUntil:load 永不完成 | 找到已有页面复用，或用 waitUntil:commit |
