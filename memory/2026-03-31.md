## MCP Servers 崩掉 + Chrome 18800 重启（19:00 CST）

### 问题根因
- 用户 18:30 手动重启 Gateway
- MCP adapter 在 Gateway 启动时同时启动 MCP servers（doubao/kimi/glm/qwen/deepseek）
- 但当时 Chrome 还没在 18800 端口运行 → 所有 MCP servers fetch 失败并崩溃
- Chrome 后来手动启动到 18800，但 MCP servers 进程已死

### 解决过程
- 手动 `openclaw browser --browser-profile openclaw start` → Chrome 稳定在 18800 ✅
- 但 Gateway mcp-adapter 没有重启已崩的 MCP servers
- 需要 `openclaw gateway restart` 让 mcp-adapter 重新启动所有 MCP servers

### 最终状态（pending 用户确认）
- Chrome 18800: ✅ 运行中
- MCP servers: ❌ 全崩（等待 gateway restart）
- 待执行: `openclaw gateway restart`（用户未确认）

### 教训
- Gateway 重启时，MCP adapter 会自动重启 MCP servers
- 但如果 Chrome彼时不在正确端口，所有 MCP servers 会同时崩
- 正确顺序：先启动 Chrome(18800) → 再重启 Gateway
- 或 Chrome 用 `--watch` 模式自动重启后，Gateway 也需要重启

## Semantic Cache reindex 500 错误修复（20:18 CST）

### 问题
- reindex 端点持续返回 500 Internal Server Error
- 搜索功能正常，但无法重建索引

### 根因
- 旧 server.py 进程启动时日志重定向到 `/tmp/semantic_cache.log`
- 该文件之后被新内容覆盖（只有旧日志）
- reindex 请求的 Python 异常被 Flask 捕获但打印到 stdout（重定向的 pipe），没有写入任何日志文件
- 实际错误被"吞掉"了

### 修复步骤
1. `ssh` 到 Ubuntu → kill 旧 server.py 进程（PID 1133991）
2. 用正确重定向重启：`nohup ~/miniconda/bin/python3 server.py > ~/semantic_cache_server.log 2>&1 &`
3. 验证：`curl http://192.168.1.18:5050/health` ✅
4. 测试 reindex：`curl -X POST http://192.168.1.18:5050/reindex?force=true` ✅

### 经验教训
- **日志重定向到 pipe 时异常可能被吞**：如果进程 stdout/stderr 重定向到 pipe（而非文件），异常输出不会持久化
- **诊断方法**：重启进程并确保日志写入文件
- **验证修复**：先测 health 端点，再测 reindex

### 涉及文件
- `/home/jet/semantic_cache/server.py` — 服务器代码
- `/tmp/semantic_cache.log` — 旧日志（已无效）
- `/home/jet/semantic_cache_server.log` — 新日志文件
