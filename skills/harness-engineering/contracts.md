# 用户做事风格 Contracts

## 来源分析

本文件分析用户(lr)与 OpenClaw 的历史聊天记录，总结出真实发生的做事风格。

---

## 行为证据

### 证据1：每件事都要有交代

**历史对话摘录**：
- LeWM 调参成功 → 凌晨心跳记录汇报
- Semantic Cache 索引 → 汇报索引量变化
- Night Build A序列 → 全部 11 个任务完成都要汇报

**行为模式**：用户会追每一个任务的完成状态，不接受"应该做了"的模糊回答。

---

### 证据2：复杂任务要分阶段确认

**历史对话摘录**：
- A序列任务：先派 subagent 调研 → 等调研结果 → 再派 subagent 执行
- 每次让 subagent 调研后汇报 → 等回复 → 再决定下一步
- MiniMax 额度重置时间点 → 用户专门问

**行为模式**：喜欢分阶段，每阶段有明确的"调研→方案→执行→验证"节奏。

---

### 证据3：喜欢多平台互相印证

**历史对话摘录**：
- "用至少 2 个平台搜索，互相印证"
- "DeepSeek + Kimi + 豆包并行搜索"
- "搜不到就标注不编造"

**行为模式**：单一来源不信任，多个平台互相印证才有说服力。

---

### 证据4：遇到问题先找方法，不要停

**历史对话摘录**：
- ESP32-Cam固件失败 → 换固件版本、重刷
- LeWM下载失败 → 换 HuggingFace 镜像、换下载方式
- MCP server崩了 → 重新获取cookie、重启顺序

**行为模式**：问题不是终点，是流程的一部分。遇到就解决，解决不了就换一个方法。

---

### 证据5：做事要有记录

**历史对话摘录**：
- "任何操作都要详细记录"
- ".learnings/ 要同步到 submodule"
- "错误记录要记录两次"

**行为模式**：
- 错误要记录：`.learnings/ERRORS.md` + `memory/YYYY-MM-DD.md`
- 重要决定要记录：写文件，不"记在脑子里"
- 产物要汇报：文件路径、链接、关键数据

---

## 用户不喜欢的回复风格

| ❌ 用户不喜欢 | ✅ 用户期望 |
|---------------|-------------|
| "好的，我来帮你做" 然后默默开始 | "已收到，预计 X 分钟" 然后定期汇报 |
| "这个问题很复杂，我正在想办法" 然后没下文 | "遇到 F2 问题，方案A不可行，重新调研方案B" |
| "根据我的分析..." 然后长篇大论没有来源 | "DeepSeek 说... + Kimi 说... + Tavily 确认..." |
| "应该没问题" | "验证通过（`node --check` 无错误输出）" |
| "我没有找到相关信息" 然后停了 | "没找到直接答案，可能的方向是 A/B/C" |

---

## 补充偏好

- **时间感知强**：喜欢有具体时间的任务（"20:00 CST"）
- **工具并行**：遇到调研任务会派多个 subagent 并行
- **文档强迫症**：代码要有注释、输出要有路径、失败要有日志
- **不喜欢等**：如果 AI 需要时间，应该先告知用户去干别的
