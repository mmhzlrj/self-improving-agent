# 小龙虾 P2P 服务交易市场 (V2)

**版本**: 2.0.0  
**设计理念**: 借鉴 Polymarket 预测市场机制，打造去中心化服务撮合平台

---

## 一、核心概念映射

| Polymarket 预测市场 | 小龙虾 V2 服务市场 |
|---------------------|-------------------|
| 预测事件 | 需求/任务 |
| 下注 Yes/No | 竞标/接单 |
| 赔率 | 服务报价 |
| 做市商 | 流动性提供者 |
| 事件结算 | 任务交付确认 |
| 奖金池 | 托管赏金 |

### 2.2 角色说明

## 四、核心机制详细设计

### 4.1 需求发布机制

#### 4.1.1 需求格式

```json
{
  "id": "req_xxx",
  "type": "service_request",
  "title": "智能门锁安装",
  "description": "需要安装小米智能门锁X，",
  "category": "安装服务",
  "tags": ["智能门锁", "小米", "上海"],
  "budget": {
    "min": 100,
    "max": 300,
    "currency": "CNY"
  },
  "deadline": "2024-12-31T18:00:00Z",
  "requirements": [
    "有智能门锁安装经验",
    "能提供发票"
  ],
  "requester": {
    "device_id": "xxx",
    "reputation_score": 4.5
  },
  "bounty": 50,
  "escrow": 200
}
```

#### 4.1.2 需求字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| title | ✅ | 需求标题（20字内） |
| description | ✅ | 详细描述 |
| category | ✅ | 分类：安装/维修/设计/咨询/其他 |
| tags | ✅ | 标签（便于搜索匹配） |
| budget | ✅ | 预算区间 |
| deadline | ✅ | 有效期/截止时间 |
| requirements | ❌ | 特殊要求 |
| bounty | ✅ | 赏金（吸引服务商） |
| escrow | ✅ | 托管金额（预付款） |

#### 4.1.3 赏金机制（类似 Polymarket 奖金池）

- **赏金作用**：吸引服务商投标，类似 Polymarket 的参与奖励
- **赏金来源**：需求方额外支付
- **分配方式**：
  - 赢家（被选中的服务商）：获得赏金 + 服务费
  - 参与者（参与投标但未中标）：按比例分剩余赏金

```
赏金分配示例（赏金 50 元，5 人投标）：
- 中标者：50 × 60% = 30 元
- 其他 4 人：50 × 40% / 4 = 5 元/人
```

### 4.2 竞价/匹配机制

#### 4.2.1 竞价方式

**方式一：明标竞价**
```
需求发布后，服务商公开报价
需求方看到所有报价，选择最优
```

**方式二：暗标竞价**
```
需求方设置预算上限
服务商匿名报价，低于预算才显示
系统匹配最优报价
```

**方式三：智能匹配**
```
需求方设置需求 + 预算
系统根据服务商评分、距离、历史交易自动推荐
需求方一键确认
```

#### 4.2.2 报价格式

```json
{
  "id": "bid_xxx",
  "request_id": "req_xxx",
  "provider": {
    "device_id": "xxx",
    "nickname": "张师傅",
    "reputation_score": 4.8,
    "completed_jobs": 128
  },
  "price": 200,
  "estimated_time": "2小时",
  "proposal": "周六上午可上门，包含一年质保",
  "timestamp": "2024-12-20T10:00:00Z"
}
```

#### 4.2.3 匹配算法

```python
def match(request, bids):
    # 权重评分
    scores = []
    for bid in bids:
        price_score = (request.budget.max - bid.price) / request.budget.max * 40
        rep_score = bid.provider.reputation_score * 30
        time_score = 30 if bid.estimated_time <= "2小时" else 15
        
        total = price_score + rep_score + time_score
        scores.append((bid, total))
    
    # 返回最高分
    return sorted(scores, key=lambda x: x[1], reverse=True)[0]
```

### 4.3 托管与结算机制

#### 4.3.1 托管流程（类似 Polymarket Escrow）

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  需求方      │     │  托管合约    │     │  服务商     │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                   │
       │ 支付 托管金+赏金   │                   │
       │───────────────────>│                   │
       │                   │ 锁定资金           │
       │<───────────────────│                   │
       │                   │                   │
       │                   │ 交付服务           │
       │                   │<───────────────────│
       │                   │                   │
       │ 确认验收/ Dispute │                   │
       │───────────────────>│                   │
       │                   │                   │
       │ 验收通过           │ 释放资金           │
       │<───────────────────│───────────────────>│
       │                   │                   │
```

#### 4.3.2 资金释放规则

| 场景 | 处理方式 |
|------|----------|
| 验收通过 | 托管金 → 服务商，赏金按规则分配 |
| 验收拒绝 | 进入 Dispute 流程 |
| 超时未确认 | 自动确认（可配置） |
| 双方取消 | 退款（扣除少量手续费） |

#### 4.3.3 Dispute 争议处理

```
┌─────────────────────────────────────┐
│           Dispute 流程              │
├─────────────────────────────────────┤
│ 1. 任何一方发起 Dispute             │
│ 2. 提供证据（聊天记录/照片/文件）    │
│ 3. 流动性提供者仲裁                  │
│ 4. 裁决：                           │
│    - 需求方胜诉：全额退款            │
│    - 服务商胜诉：全额支付            │
│    - 折中方案：各退一步              │
└─────────────────────────────────────┘
```

### 4.4 信用体系

#### 4.4.1 评分系统

```json
{
  "user": "device_xxx",
  "reputation": {
    "score": 4.7,
    "total_reviews": 50,
    "as_requester": {
      "count": 30,
      "avg_rating": 4.8
    },
    "as_provider": {
      "count": 20,
      "avg_rating": 4.6
    }
  },
  "badges": [
    "靠谱雇主",
    "手艺精湛",
    "速度担当"
  ],
  "verified": true
}
```

#### 4.4.2 信任增强机制

| 机制 | 说明 |
|------|------|
| **押金制** | 服务商需缴纳押金才能接单 |
| **保证金** | 流动性提供者需缴纳保证金 |
| **芝麻信用** | 接入外部信用体系 |
| **熟人背书** | 邀请好友，好友可为你担保 |
| **历史记录** | 公开透明的全部交易记录 |

#### 4.4.3 评分权重

```
最终评分 = 交易评分 × 70% + 互评 × 20% + 芝麻信用 × 10%
```

### 5.3 核心模块

| 模块 | 功能 | 技术选型 |
|------|------|----------|

| **存储层** | 需求/报价/评价存储 | SQLite + IPFS |
| **身份认证** | 设备绑定、签名 | Ed25519 |

## 六、与原 P2P 聊天的区别

### 6.1 功能对比

| 功能 | V1 纯聊天 | V2 服务市场 |
|------|----------|-------------|
| 核心功能 | 点对点消息 | 需求发布 + 竞价 + 托管 |
| 角色 | 单一（用户） | 三重（需求方/服务商/流动性） |
| 连接方式 | 直接连接 | 可直连，可通过引导节点 |
| 消息类型 | 纯文本 | 结构化需求 + 报价 + 评价 |
| 资金安全 | 无 | 托管 + Dispute |
| 信任机制 | 无 | 信用评分 + 押金 |
| 搜索匹配 | 无 | 标签 + 智能推荐 |
| 激励机制 | 无 | 赏金 + 评价奖励 |

### 6.2 架构升级

```
V1:  直接连接 → 发送消息 → 断开
     (简单直接，无中间态)

V2:  发布需求 → 广播 → 竞价 → 托管 → 交付 → 结算 → 评价
     (完整交易生命周期)
```

### 6.3 用户体验变化

| 维度 | V1 | V2 |
|------|----|----|
| 首次使用 | 交换 IP 地址 | 启动即可（引导节点自动发现） |
| 找服务商 | 手动询问 | 搜索 + 智能推荐 |
| 谈价格 | 来回聊天 | 竞价 + 明码标价 |
| 付款 | 线下转账 | 托管 + 自动结算 |
| 信任 | 无保障 | 评分 + 押金 |

## 十、总结

小龙虾 V2 从**纯聊天工具**升级为**去中心化服务交易市场**：

1. **借鉴 Polymarket**：赏金吸引、竞价机制、托管结算
2. **保持 P2P 核心**：不依赖服务器，点对点直接交易
3. **信任机制**：信用评分 + 押金 + Dispute
4. **平滑过渡**：V1 用户可升级，保留原有功能
