# Deep Research 使用规范

> 本文档由 2026-03-24 教训总结。MiniMax 做系统性调研时多次违反规范，导致结果质量不达标。

## 核心原则

**Deep Research 不是"问一遍就完"**。它是多轮迭代、多工具协同的深度调研流程。

## 第一步：工具状态检查（必须做，不能跳过）

> **详细操作步骤见独立 SOP 文件：** `~/.openclaw/workspace/.learnings/deep-research-tool-check-SOP.md`
>
> 该 SOP 极度详细，包含：完整命令、常见错误表、判断标准、记录格式。必须按 SOP 执行，不允许跳过任何步骤。

**每次启动调研前，必须逐个确认每个工具的实际可用性。**

### ⚠️ 核心原则：测试 MCP Server 本身，不是测试页面能不能打开

**错误做法**（MiniMax 犯过的错误）：
- ❌ 用 Playwright 打开 5 个 AI 网页版，看到页面加载就认为"检查通过"
- ❌ 检查页面标题来判断登录态（标题在登录和未登录时可能一样）
- ❌ 用 `textContent('body')` 搜索"登录"文字来判断（文字可能出现在无关位置）
- ❌ 只检查 Chrome 能不能连，不检查 MCP Server 能不能调通

**正确做法**：
- ✅ **直接调用 webauth 工具本身**（`doubao_doubao_chat`、`kimi_kimi_chat` 等），看能不能收到回复
- ✅ **直接调用搜索工具本身**（`tavily_search`、`minimax search`），看能不能返回结果
- ✅ **记录每个工具的响应时间**，异常慢的也要标记

### 检查清单（按顺序执行）

#### 1. Chrome CDP（5个 webauth 工具的基础依赖）

```bash
curl -s --max-time 3 "http://127.0.0.1:9223/json/version"
```

- ✅ 返回 JSON 包含 `Browser` 字段 → 正常
- ❌ 连接超时或无响应 → Chrome-Debug-Profile 未启动，问用户是否需要启动
- ⚠️ Chrome 在但 5 个 webauth 工具都超时 → 可能是 Gateway 与 Chrome 之间连接问题

#### 2. webauth 5平台（直接调用工具本身，不是打开页面）

**测试方法**：给每个平台发一个简单问题，能收到回复就说明 MCP Server + 登录凭证都正常。

```
并行调用（同时发出，等待所有返回）：
1. doubao_doubao_chat("你好，回复OK") → ✅/❌ + 响应时间
2. kimi_kimi_chat("你好，回复OK") → ✅/❌ + 响应时间
3. glm_glm_chat("你好，回复OK") → ✅/❌ + 响应时间
4. qwen_qwen_chat("你好，回复OK") → ✅/❌ + 响应时间
5. deepseek_deepseek_chat("你好，回复OK") → ✅/❌ + 响应时间
```

**判断标准**：
- ✅ 收到正常回复（非空、非错误信息）→ 工具正常
- ❌ 超时（>30秒无响应）→ MCP Server 或登录凭证可能过期
- ❌ 返回错误信息（如 `MCP error -32001`、`bad request`）→ 需要修复

**不可用时的排查步骤**：
1. 先确认 Chrome CDP 可用（步骤1）
2. 如果 Chrome 可用但工具超时 → 可能是 Token/Cookie 过期
3. 检查各 MCP Server 的日志：`~/.openclaw/extensions/*-mcp-server/`
4. 问用户：是否需要重新登录或刷新凭证？

#### 3. 搜索工具（直接调用，不是打开网页）

```
并行调用：
1. python3 ~/.openclaw/workspace/skills/minimax-tools/minimax.py search "test"
   → ✅ 返回搜索结果 / ❌ 报错

2. python3 ~/.openclaw/workspace/skills/tavily-search/search.py "test"
   → ✅ 返回搜索结果 / ❌ 报错

3. web_search（Brave）→ 已知无 API Key，跳过（用 minimax search 替代）
```

#### 4. subagent

```
sessions_spawn(task="回复OK", runtime="subagent", mode="run", runTimeoutSeconds=30)
→ ✅ 收到回复 / ❌ 超时或报错
```

### 检查结果记录格式

```
工具状态检查（YYYY-MM-DD HH:MM）：
- Chrome CDP 9223: ✅ Chrome/版本号
- Doubao: ✅ (3.2s) / ❌ 超时/错误信息
- Kimi: ✅ (2.8s) / ❌ 超时/错误信息
- GLM: ✅ (4.1s) / ❌ 超时/错误信息
- Qwen: ✅ (3.5s) / ❌ 超时/错误信息
- DeepSeek: ✅ (5.2s) / ❌ 超时/错误信息
- minimax search: ✅ (1.5s) / ❌ 错误信息
- tavily: ✅ (1.8s) / ❌ 错误信息
- subagent: ✅ (4s) / ❌ 超时/错误信息
```

**注意**：记录响应时间很重要。如果一个工具突然变得很慢（比如从 3s 变成 20s），虽然还没超时，也应该标记 ⚠️ 并告知用户。

### 重要规则

- **发现工具不可用 → 立刻停下来问用户**："XXX 工具不可用，是要先修复还是直接用能用的工具继续？"
- **绝对禁止**：发现工具不可用但不告诉用户，偷偷用不完整的工具调研
- **绝对禁止**：用 Playwright 打开页面代替直接调用工具来"检查"状态

## 第二步：理解调研范围

### 拆解调研任务

把大任务拆成小问题，每个小问题独立调研。

**示例**：调研 ROBOT-SOP 的 10 个章节
- ❌ 错误做法：派一个 subagent 说"去调研所有章节"
- ✅ 正确做法：逐章列出具体问题，每个问题独立搜索

### 每个问题需要验证的维度

| 维度 | 具体检查 |
|------|---------|
| **真实度** | 技术参数、价格、发布时间是否准确？有无过时信息？ |
| **详细度** | 描述是否足够详细？有没有遗漏的关键参数/步骤/成本？ |
| **更优方案** | 有没有比文档中方案更好/更便宜/更新的替代方案？ |
| **盲区** | 有哪些用户应该知道但文档没写的重要事项？ |

## 第三步：多工具并行搜索（至少 2 轮）

### 第一轮：广度搜索

同时使用多个工具从不同角度搜索：

```
并行启动：
1. web_search "关键词 + 最新"  → 搜索最新信息
2. deepseek_deepseek_chat "具体问题"  → AI 深度分析
3. doubao_doubao_chat "具体问题"  → 另一个 AI 视角
4. kimi_kimi_chat "具体问题"  → 第三个 AI 视角
5. web_fetch "官方文档URL"  → 获取权威来源
```

### 第一轮完成后：分析结果

- 哪些信息多个来源一致？→ 高可信度
- 哪些信息只有一个来源？→ 需要进一步验证
- 有没有新的关键词/方向值得深入？→ 触发第二轮

### 第二轮：针对性深入

针对第一轮发现的不确定点和新方向：

```
针对不确定信息：
1. web_fetch "具体页面"  → 获取原文验证
2. 用不同的 web_search 关键词  → 交叉验证
3. 再问一个 AI 平台 "这个信息对吗？最新是什么？"  → 第三方验证
```

### 迭代条件

**什么时候继续搜：**
- 发现信息不一致（两个来源说法矛盾）
- 发现新的重要方向没有覆盖
- 有具体数据需要验证（价格、日期、规格等）

**什么时候停止：**
- 多个来源信息一致
- 已经验证了关键数据
- 搜索结果开始重复，没有新信息

**绝对禁止**：只搜一轮就出结论。这是今天最大的错误——NemoClaw 漏查就是因为搜了一轮就说"没找到官方确认"。

## 第四步：Chrome + webauth 工具的正确使用

### webauth 5平台适合的场景

| 场景 | 是否适合 | 原因 |
|------|---------|------|
| 短问答（<10秒） | ✅ 适合 | 快速获取 AI 观点 |
| 技术概念解释 | ✅ 适合 | AI 擅长解释概念 |
| 长调研（>10秒） | ❌ 不适合 | 容易超时 |
| 搜索最新价格/发布时间 | ⚠️ 谨慎 | AI 的信息可能过时 |

### 重要：webauth 工具不是搜索引擎

- webauth 是**问 AI**，不是**搜索网页**
- AI 可能说"没找到"，但 web_search/web_fetch 能找到
- **验证关键数据时，必须用 web_search + web_fetch**，不能只问 AI

### Chrome CDP 的检查

```bash
# 检查 Chrome 是否运行
curl -s --max-time 3 "http://127.0.0.1:9223/json/version"

# 检查各平台是否登录（用 Playwright）
node -e "
const {chromium} = require('playwright');
(async () => {
  const browser = await chromium.connectOverCDP('http://127.0.0.1:9223');
  const ctx = await browser.newContext();
  const page = await ctx.newPage();
  await page.goto('https://www.doubao.com/chat/', {timeout: 8000});
  console.log('Title:', await page.title());
  await browser.close();
})();
"
```

## 第五步：输出格式

### 每个问题的输出模板

```markdown
## [问题名称]

### ✅ 验证通过
- [内容] → 来源：[URL 或 AI 平台]

### ⚠️ 需要更新
- [原文内容] → 实际应该是：[正确内容] → 来源：[URL]

### 💡 更优方案
- [新方案] → 来源：[URL]
- 对比：[原方案 vs 新方案的优劣]

### 🔍 盲区发现
- [文档没提到的重要事项] → 来源：[URL]
```

### 重要：标注来源

每个发现都必须标注来源，不能说"根据我的知识"或省略来源。

**错误示例**：
> "NemoClaw 没找到官方确认"

**正确示例**：
> "通过 web_search 'NemoClaw NVIDIA GTC 2026' 搜索前 10 条结果，未发现 NVIDIA 官方发布页面。再通过 deepseek_deepseek_chat 询问，回复说'未找到相关信息'。但可能搜索关键词不够准确，建议进一步用 'OpenShell NVIDIA' 等关键词搜索。"

## 第六步：汇报给用户

### 汇报规范

1. **完整贴出新内容**，不省略任何描述
2. **改完文件立即 mdview 打开**给用户看
3. **列出所有建议修改**，等用户确认后再改文件
4. **commit 只在用户确认后**才执行
5. **预估等待时间**：在开始调研前告知用户预计耗时

### 预估等待时间

| 调研规模 | 预估时间 | 说明 |
|---------|---------|------|
| 单个小问题 | 2-5 分钟 | 1-2 轮搜索 |
| 1 个章节（5-10 个小点） | 10-20 分钟 | 每个小点 2 轮搜索，串行执行 |
| 多个章节（20+ 个小点） | 30-60 分钟 | 建议 subagent 并行 |
| 全文档系统性调研（10 章） | 60-120 分钟 | 分批执行，每批汇报一次 |

**预估公式**：
```
总时间 ≈ 小点数量 × 每点 2 轮 × 每轮 2 分钟
```

例：20 个小点 → 20 × 2 × 2 = 80 分钟 ≈ 1.5 小时

**实际记录**：每次调研完成后，在调研报告中记录实际耗时，用于优化未来预估。

### 汇报格式（生成汇总 MD）

调研完成后，生成一个**汇总修改意见 MD 文件**，用 `mdview --review` 打开，让用户快速审批。

## 常见错误（避免重犯）

| 错误 | 正确做法 |
|------|---------|
| 派一个 subagent 调研所有内容 | 拆分成小问题，每个独立调研 |
| 只问 AI 不搜索网页 | 关键数据必须 web_search + web_fetch 验证 |
| 搜一轮就出结论 | 至少 2 轮，发现不一致继续搜 |
| 工具不可用不告诉用户 | 停下来问：修还是用能用的继续 |
| 汇报省略细节 | 完整贴出新内容 |
| 直接修改文件 | 列出建议，等用户确认后再改 |
| 用户问"你知道怎么用吗"时不读文档就回答 | 必须先 `read` 文档再回复 |
| 工具状态标记⚠️但不实际测试 | 每个工具都必须实际执行测试命令 |
| 凭记忆回答而不重新读文档 | 新会话 = 新开始，必须重新 read |

## 2026-03-24 教训记录

### 事件
用户要求系统性调研 ROBOT-SOP 10 个章节的真实性。

### MiniMax 犯的错误
1. 没有先检查工具状态就启动调研
2. NemoClaw 调研只搜了一轮就结论"没找到官方确认"
3. 没有使用 web_search / web_fetch 验证关键数据
4. 没有 Chrome CDP 连接来检查 webauth 工具状态
5. subagent 只跑了 3 分钟就输出完整报告，调研深度不够

### 用户评价
> "你是不是偷懒没用上deep-search 的所有工具？"

### 根本原因
- subagent 被设计为"一次性完成所有章节"，工作量太大
- 没有要求每章独立验证、多轮迭代
- 没有要求标注具体来源 URL
- 时间太短（3分钟做 10 章调研 = 平均每章 18 秒，明显不够）

### 改进措施
- 本文档即为改进措施
- 调研前必须检查工具状态
- 每个问题至少 2 轮搜索
- 关键数据必须标注 URL 来源
- 汇报时列出修改建议，等用户确认后再改文件
