# 第一章：概念与愿景

## 1.1 项目愿景

**0-1** —— 不是一台机器，是你人生的另一面。

| 层含义 | 说明 |
|--------|------|
| **机器层** | 二进制的起点，代表硅基生命的底层语言 |
| **制造层** | 第一次「从零到一」用 3D 打印制造一个完整的机器人 |
| **生命层** | 人一生的「0 到 1」——从出生到结束，机器的记忆就是我的人生存档 |

0-1 不是一个固定的产品，而是一个会跟着 AI 能力一起长大的伙伴。10 年分五个阶段，每一年，期望和能力都在共同进化。

> 「0-1」还很适合用来做机器人面部的默认表情，且名字简单易记，未来易拓展至「1-2」「2-3」等。

## 1.2 核心系统：贵庚

**贵庚** 是 0-1 的记忆系统，专为保存一个人完整的一生而设计。

- **贵** = 粤语里对他人的敬称，也暗含「珍贵」
- **庚** = 天干第七位，对应时间、年龄、周期——记忆本质上就是时间的切片
- **贵庚** 在粤语里是问老人家年龄的敬语，带着一种对岁月沉淀的敬畏

> 贵庚存的不是数据，是一个人的年龄和尊严。

### 核心价值主张

| 价值 | 说明 | 优先级 |
|------|------|--------|
| **记忆延续（贵庚）** | 帮你记录身边发生的事情，随时回顾 | P0 |
| **陪伴对话** | 随时语音交流，有问必答 | P0 |
| **执行任务** | 帮你完成简单的物理任务 | P1 |
| **安全守护** | 身份认证、防抢防盗、紧急响应 | P1 |
| **数据隐私** | 所有数据本地处理，不传云端 | P0 |

### 贵庚记忆系统技术架构

> ⚠️ **0-1 当前状态**：本节为新增，描述贵庚记忆系统的技术路线。阶段一（SOP Phase 1-4）以日志式记录为主，阶段二（Phase 5-6）开始接入完整记忆架构。

#### 核心定位：原生数据的容器，而非技术路线

**贵庚最重要的功能，不是某一种技术实现，而是完成原生数据的积累。**

**起源**：贵庚以 OpenClaw 内置的记忆系统（MEMORY.md + 日志体系）为基础诞生，随着项目推进和数据积累，再积极拥抱其他开源记忆架构，或根据私人定制需求自己打造专属的记忆引擎。技术选型始终是开放的。

**目标**：数据积累到一定程度后，可以自由切换到更先进的方案——无论是开源社区的最新成果，还是自研的个性化实现。

**硬件偏好**：未来采购新设备时，优先选择带 NPU 的硬件，并重点关注能够充分利用 NPU 进行本地推理的记忆架构（现有 OpenClaw + 未来自研）。其次，优先选择统一内存架构的大内存设备（如 AMD AI Halo 128GB、苹果 M 系列、DGX Spark 等），大容量统一内存可以让整个记忆系统在设备端完整运行，无需依赖云端。

新的硬件会诞生，新的算法会出现，新的架构会迭代——每一个时代都会有当时最优的技术路线。但这些技术路线变化越快，越说明**积累原始数据这件事本身的珍贵**。

贵庚的使命是：从现在开始，把身边最真实的数据持续地、完整地存下来。在任何一个新的技术时代来临的时候，都能用当时最好的方法，重新解析这些历史数据，从中获得新的理解。

```
今天的录像 → 未来某天用新算法重新理解 → 发现今天没看到的东西
今天的对话 → 未来某天用新模型重新分析 → 读出今天没读出的情绪
```

**技术会过时，数据不会。**

这不是"先用 RAG，以后再换"的技术路线规划——而是**以数据积累为核心，分阶段选择最合适的技术工具**。阶段一用 RAG 是因为 RAG 够用、够快、能立刻产生价值；阶段二换边权重模型是因为积累了足够多的真实数据，需要更深的个性化理解。无论哪条路线，**原始数据始终以最完整的形态保留**。

#### 发展阶段

| 阶段 | 时间 | 数据来源 | 核心能力 |
|------|------|---------|---------|
| **前期** | 阶段一（1-2年）| 文本对话 + 语音 | 学语言习惯、记住重要内容 |
| **中期** | 阶段二（3-4年）| 全景摄像头 7×24 小时录像 | AI 分析标注视频，建立生活行为时间线 |
| **长期** | 阶段三+（5年+）| 持续扩展数据源 | 多模态理解、因果推理、个性化生成 |

> **每个阶段的技术选型都可能不同，但原始数据的存储格式始终保持开放和完整。**

#### 三个核心设计难点（技术层面）

这三个问题描述的是**技术实现层面**的具体挑战，是方法论层面的问题，而非目的层面的约束。

| 难点 | 技术挑战 | 解决思路 |
|------|---------|---------|
| 如何理解「为什么重要」 | 重要性权重建模 | 边权重实例型记忆，交互频率+事件权重自动调整 |
| 如何建立时间线和因果关系 | 时序索引与图结构 | 时序事件图谱，带时间戳和因果边 |
| 如何保持「不唠叨」的分寸感 | 主动介入策略 | OpenClaw RL 强化学习，从真实交互中学习 |

#### 模型路线：边权重实例型大模型

**概念**：区别于通用大模型的参数化知识（知识存在模型权重里），边权重实例型大模型将用户的个性化记忆以"实例权重"的形式附加在通用基座模型之上——通用能力不动，只调用户的专属记忆权重。

```
通用基座（如 Qwen3-7B）
    +
用户记忆实例权重（边权重层）
    =
贵庚专属模型（能记住你、理解你、为你定制）
```

**与 RAG 的区别**：

| 维度 | RAG（检索增强生成）| 边权重实例型模型 |
|------|-------------------|-----------------|
| 存储形式 | 外部向量数据库，每次检索 | 实例权重，模型内化 |
| 响应速度 | 检索有延迟（毫秒级）| 无检索延迟 |
| 个性化程度 | 依赖检索质量 | 实例权重直接改变模型行为 |
| 隐私 | 数据在外部存储 | 数据编码进权重（可加密）|
| 训练成本 | 低（无需训练）| 中（需要 RLHF + 实例微调）|

**阶段一 MVP 方案（RAG 过渡）**：
- 用向量数据库（Qdrant/Chroma）存储记忆片段
- 每次对话前检索相关记忆注入上下文
- 同步记录结构化日志（时间、事件、人物、地点）
- **优点**：零训练成本，立刻能用
- **缺点**：检索质量依赖 embedding 模型，无法真正"理解"记忆权重

**阶段二目标（边权重实例型）**：
- 在阶段一积累的真实数据上，用 OpenClaw RL 做实例权重训练
- 基座模型：Qwen3.5-14B 或更大（取决于 AMD AI Halo 128GB 是否到位）
- 用 GRPO + OPD 混合方法，让模型从对话历史中学习什么重要、什么该记住

#### OpenClaw RL 集成

**OpenClaw RL**（GitHub: Gen-Verse/OpenClaw-RL）是一个将对话转化为训练信号的强化学习框架，包含三个优化方法：

| 方法 | 原理 | 适合场景 |
|------|------|---------|
| **GRPO（二值 RL）** | Process Reward Model 对每个对话轮次打分 | 判断"这次回答好不好" |
| **OPD（离线策略蒸馏）** | Judge 模型提取文本线索辅助训练 | 从历史对话中提取隐式偏好 |
| **Combination** | GRPO + OPD 混合，优化的信号更强 | 复杂对话场景 |

**对贵庚的价值**：

1. **训练"闭嘴"的分寸感**：用 RL 让模型学习什么时候该主动介入、什么时候该沉默——这不是规则能写死的，需要从真实交互中学习
2. **实例级个性化**：用 OPD 方法从用户的历史对话中提取隐式偏好，转化为实例权重
3. **异步训练架构**：Agent 服务、Rollout 收集、PRM 评估、策略训练四个组件独立运行，不影响实时对话

**部署路径**：

```
阶段一：OpenClaw 默认对话（无 RL）→ 积累对话日志
阶段二：接入 OpenClaw RL，在本地 GPU 上训练实例权重
        - 基座：Qwen3.5-14B（AMD AI Halo 128GB）
        - 训练集：阶段一积累的真实对话
        - 方法：GRPO + OPD Combination
阶段三：贵庚专属模型本地部署，实时推理
```

> 📌 **参考**：OpenClaw RL Skill 路径 `skills/openclaw-rl/`，技术报告：https://arxiv.org/abs/2603.10165

#### RynnBrain 与贵庚记忆系统的接入位置

> **关键澄清**：RynnBrain 是一个**视觉-语言推理模型**（基于 Qwen3-VL 的具身基座），不是独立的记忆存储系统。它的"时空记忆"能力实际上是**在视频中定位/追踪物体 + 推理时序关系**。

**RynnBrain 的核心能力**（官方 Demo，2026-02）：

| 能力 | 说明 | 对贵庚的价值 |
|------|------|------------|
| **时空定位** | 给定视频 + 自然语言描述 → 返回物体 bounding box + 时间戳 | 回答"上周我把钥匙放哪了" |
| **轨迹追踪** | 在视频流中追踪物体随时间的位置变化 | 记录物品移动路径 |
| **交错推理** | 文本推理和空间定位交替进行 | "昨天在厨房拿的东西现在在哪" |
| **Affordance** | 理解物体"可以用来做什么" | 推理用户意图 |

**RynnBrain 的记忆更新机制**（来源：arXiv 论文 Section 4）：

| 记忆类型 | 实现方式 |
|---------|---------|
| **短期工作记忆** | 多轮对话格式，交互历史作为显式缓冲区保留 |
| **长期经验记忆** | 预训练内化（1990万样本），以模型参数形式固化 |
| **防止灾难性遗忘** | GRPO 训练 KL 散度惩罚（β=0.02，仅限 RynnBrain-CoP 变体），新记忆不覆盖旧记忆 |
| **持续学习** | 连续100个任务微调后，基础能力保留率 94%（来源：arXiv:2602.14979 附录持续学习实验 + RynnBrain 官方 GitHub 技术白皮书 + 2026-02-10 开源发布会披露） |

> **重要澄清**：RynnBrain 基础模型（2B/8B/30B MoE）使用标准 SFT 预训练，**不用 GRPO**。GRPO 仅用于 **RynnBrain-CoP**（Chain-of-Point 推理变体）Post-training 对齐阶段（arXiv:2602.14979 Section 4.2.1）。RynnBrain 没有公开具体的滑动窗口大小或衰减系数等参数细节，需要实测确认。

**RynnBrain-CoP GRPO 核心超参数**（来源：arXiv:2602.14979 Section 4.2.1，已核实）：

| 参数 | 配置值 | 说明 |
|:---|:---|:---|
| 组大小（group_size）| 5 | 同一 prompt 并行生成采样数，替代 PPO critic |
| KL 散度系数（β）| **0.02** | 约束策略更新幅度 |
| 裁剪范围（clip_range）| [0.2, 0.28] | 限制单次更新最大幅度 |
| 训练轮次（epoch）| 10 | GRPO 对齐完整迭代 |
| 全局批量大小 | 128 | 单轮参数更新总样本数 |
| 最大序列长度 | 16,384 tokens | 适配第一视角长视频 |
| 学习率调度 | 余弦 + 3% warmup | 线性预热后余弦衰减 |
| Rollout 推理引擎 | SGLang | 高效生成采样轨迹 |
| 视频帧采样 | 2 FPS（最大 2048 帧）| 长时程自我中心视频 |
| 奖励函数 | Fréchet Distance + Chamfer 距离 + 格式合规奖励 | 空间精度 + 格式约束 |

**RynnBrain 不是替代贵庚，而是增强感知层**：

```
贵庚记忆系统（存储+索引）
    ↑
    ← RynnBrain 输出结构化数据
    ← 现有向量检索（Chroma/PGVector）
    ← OpenClaw 日志体系
    ↑
感知输入源（RynnBrain 的视频处理）
```

|------|---------------|

**为什么不是阶段一**：
- 阶段一的核心任务是**积累数据**，RynnBrain 是处理引擎，没有足够数据时接入没有意义
- 阶段一用 OpenClaw 日志体系做简单记录，零成本、不引入额外复杂度
- 阶段二有足够多的视频数据后，RynnBrain 才能发挥最大价值

**为什么不是阶段三**：
- 阶段三的贵庚已经在用边权重实例型模型，需要的是更深的个性化理解
- RynnBrain 是感知引擎，不是记忆架构，阶段三的重点是**怎么用记忆**，不是**怎么获取记忆**

> **总结：RynnBrain 的最佳接入点是阶段二，作为 Layer 2 图谱的感知增强引擎，把视频中的时空关系转化为结构化数据注入贵庚记忆索引。**

#### 贵庚记忆索引架构：多维度检索

> **核心问题**：如何回答"你上次提这件事是什么时候来着？"——这需要多维度索引同时支持时间线回溯和语义检索。

**三层索引结构**：

| 层级 | 类型 | 解决的问题 | 工具选型 |
|------|------|-----------|---------|
| **Layer 1** | 向量索引（语义检索）| "有没有说过类似的话" | Chroma / PGVector |
| **Layer 2** | 知识图谱（实体关系）| "提到了哪些人/地点/概念" | **Mem0 OSS + Neo4j 5.x**（Mem0 自动抽取实体关系写入 Neo4j，互补非竞争；TimeTree 已退役） |
| **Layer 3** | 时序索引（时间线）| "上次是什么时候" | **Neo4j 5.x + APOC**（预计算时间字段 + B+ 树索引，替代 TimeTree；备选：原生 datetime + WHERE 范围查询） |

**每个记忆条目必须包含的基础字段**：

```json
{
  "memory_id": "uuid",
  "content": "用户提到想去日本旅游",
  "entities": [
    {"type": "person", "name": "用户"},
    {"type": "location", "name": "日本"},
    {"type": "concept", "name": "旅游"}
  ],
  "timestamp": "2026-03-20T14:30:00Z",
  "importance": 0.8,
  "embedding": [0.123, -0.456, ...],
  "last_accessed": "2026-03-24T10:00:00Z",
  "access_count": 3
}
```

**多维度检索场景**：

| 检索维度 | 实现方式 | 示例查询 |
|---------|---------|---------|
| **按时间** | `timestamp` + Neo4j 原生时间函数/APOC | "上周说过什么？" |
| **按人物** | NER 提取 → 实体节点边 | "关于XXX上次说了什么" |
| **按地点** | NER 提取 → Location 实体 | "我在家里提过这件事吗" |
| **按语义** | embedding 余弦相似度 | "有没有说过类似的" |
| **按重要性** | `importance` 权重排序 | "最重要的那些记忆是什么" |

**参考案例：Windows 11 Recall 的检索架构**

微软 Windows 11 Recall（2024）是目前最接近贵庚检索目标的产品级实现，其架构值得直接借鉴：

```
┌─────────────────────────────────────────────┐
│  Snapshot Service（后台捕获进程）             │
│  每隔几秒截图 → 加密存入 Snapshot Store     │
├─────────────────────────────────────────────┤
│  VBS Enclave（硬件级安全隔离区）             │
│  ├─ Semantic Index（向量索引）              │
│  ├─ Secure Settings（加密配置）             │
│  └─ Snapshot Store（加密截图库）            │
├─────────────────────────────────────────────┤
│  Recall UI（用户界面，在 Enclave 外）        │
│  时间线 + 自然语言搜索 + 快照查看            │
└─────────────────────────────────────────────┘
```

| Recall 组件 | 功能 | 对贵庚的借鉴 |
|-----------|------|------------|
| **Semantic Index** | 截图+文本转向量，NPU 推理，本地完成 | 贵庚 Layer 1 向量检索 ✅ |
| **Snapshot Store + 元数据** | 时间戳、应用名、标题栏、停留时长 | 贵庚 raw data + 结构化日志 ✅ |
| **时间线导航** | 按时间回溯浏览所有快照 | 贵庚 Layer 3 时序索引 ✅ |
| **VBS Enclave** | 硬件级隔离，数据不离本地，密钥 TPM 保护 | 贵庚缺少 ⚠️ 需补充 |

**贵庚 vs Recall 的核心差距**：Recall 有 VBS Enclave 做硬件级隐私隔离，数据即使被物理访问也需要 Windows Hello 才能解密。贵庚的 NAS 数据在静态存储时是否有同等保护？密钥如何管理？**这是存储安全层面需要补充的设计。**

**检索延迟分析（"去年这个时候我们在干嘛"）**：

```
① 自然语言解析
"去年这个时候" → 时间范围 [2025-03-24 00:00, 2025-03-24 23:59]
        ↓
② Layer 3 时序索引（PostgreSQL B-tree on timestamp）
命中：2025-03-24 的所有记忆条目 → ~N 条
        ↓
③ Layer 1 向量语义检索（并行）
在 N 条中做 embedding 余弦相似度匹配
        ↓
④ Layer 2 实体过滤（可选）
过滤人物/地点实体，进一步缩小范围
        ↓
⑤ 合并 + 重要性排序
返回 Top-K 结果
```

**预估延迟**：步骤 ②（时序过滤）在 B-tree 索引上约 1-5ms；步骤 ③（向量检索 1M 向量）在 HNSW 上约 10-50ms；步骤 ④-⑤ 在内存中毫秒级完成。**索引阶段端到端约 50-100ms**（不含 LLM 生成），在不涉及冻层读取的情况下完全可以接受。**含 LLM 生成的完整响应延迟约 1-3 秒**。

**贵庚检索实现：两阶段融合排序（参考 Recall）**

参考 Recall 的两阶段检索架构，贵庚的查询链路设计如下：

```
第一阶段：候选集生成
├─ 语义路径：查询 → Embedding → 向量索引 Top-K（如 K=200）
└─ 时间路径：解析"去年3月" → PostgreSQL timestamp B-tree → 时间窗口内快照

第二阶段：重排序
候选集 → 轻量级交叉编码器 → 相关度打分 → Top-20 返回
```

**贵庚回答"上次是什么时候"的核心**：查询时 `ORDER BY timestamp DESC LIMIT 1`，结合 `last_accessed` 追踪记忆被调用频率，自动提升高频调用的重要性权重。

**工具推荐**：

| 工具 | 角色 | 说明 |
|------|------|------|
| **Mem0** | Graph + Vector 核心引擎（应用层） | 专为 AI Agent 设计，LLM 自动抽取实体关系，支持 Neo4j 作为图后端；Mem0 负责"记什么"，Neo4j 负责"怎么存怎么查"，互补非竞争 |

**阶段一路线（轻量 MVP）**：
- OpenClaw MEMORY.md 日志体系 → 定期结构化导出
- PostgreSQL + PGVector 存储，embeding 由本地模型生成
- **优点**：零新增工具，立刻能用
- **缺点**：无自动实体提取，需手动打标签

**阶段二路线（完整架构）**：
- Mem0 作为记忆核心层，统一管理向量+图+时序
- Neo4j 作为关系查询增强，支持复杂推理
- AI 辅助实体提取（Whisper 音频→NER 实体识别→自动打标签）
- **优点**：完整多维度检索能力
- **缺点**：需要一定的运维成本

#### 检索质量反馈闭环：自我提升机制

> 检索结果好不好，不能靠贵庚自己判断——要让用户来说。

**触发方式**：每次检索回答后，用户可以直接说"没找对"/"不对"/"贵庚记错了"，贵庚立刻记录这次错误，转入自我修正流程。

**反馈闭环链路**：

```
用户："没找对，上次不是说想去云南吗"
        ↓
贵庚记录错误：
├─ 错误类型：检索漏召回 / 相关性排序错误 / 时间范围错误
├─ 错误内容：当时问的是什么 → 贵庚的回答是什么
├─ 用户纠正：用户补充了什么
        ↓
修正记忆条目：
├─ 补充遗漏的实体/时间戳/重要性权重
├─ 记录这次错误到反馈日志
        ↓
触发再检索：用纠正后的上下文重新检索，返回结果
```

**类比 OpenClaw Self-Improving Agent 机制**：

| OpenClaw self-improving | 贵庚记忆检索 |
|----------------------|------------|
| 命令执行失败 → 记录错误 | 检索错误 → 记录错误 |
| 用户纠正 → 记录纠正 | 用户纠正 → 记录纠正 |
| 更新到 .learnings/ERRORS.md | 更新到记忆反馈日志 |
| 复盘改进 | 重新检索 + 更新索引权重 |

**反馈日志的格式**：

```json
{
  "feedback_id": "uuid",
  "timestamp": "2026-03-24T21:30:00Z",
  "query": "上次说想去云南是什么时候",
  "retrieval_result": "找到2条，但漏掉了关键对话",
  "user_correction": "补充：2026-03-18晚餐时讨论云南行程",
  "error_type": "漏召回",
  "corrected_entities": ["云南", "2026-03-18", "行程计划"],
  "index_update": {
    "memory_id": "原始记忆ID",
    "action": "补充标签",
    "entities_added": ["云南", "行程"]
  }
}
```

**反馈驱动的索引优化**：

每次反馈 → 自动调整索引权重：
- 某条记忆被反馈"漏召回" → importance 加权 +50%
- 某个实体的检索命中频率低 → embedding 降权，下次减少暴露
- 用户反复纠正同一类问题 → 记录为模式识别触发点

> **核心原则**：贵庚不是完美的，它的每一次错误都是数据积累的一部分。用户纠正错误时，不仅在修正单次检索，还在优化整个记忆系统的质量。

#### 开源标注工具链

> 贵庚积累的原生数据（视频/音频/图像/文本）需要系统化标注，才能在检索时精准命中。不同数据类型使用不同工具，全部本地部署，数据不上云。

**推荐组合**：

| 工具 | 最适合 | 支持类型 | 本地部署 |
|------|--------|---------|---------|
| **Label Studio** | 全类型统一平台 | 视频✅ 音频✅ 图像✅ 文本✅ | ✅ Docker |
| **VIA (VGG)** | 前期快速采集 | 视频✅ 音频✅ 图像✅ | ✅ 纯前端，双击即用 |
| **X-AnyLabeling** | 视频/图像 AI 辅助标注 | 视频✅ 图像✅ | ✅ Pip 安装 |
| **Argilla** | 对话/文本标注 | 文本✅ 音频✅ | ✅ Docker |
| **Praat** | 语音精度标注 | 音频✅（音素/时间戳）| ✅ 桌面应用 |

**推荐理由**：
- **Label Studio**：全类型覆盖，生态最完整，Docker 一键部署，个人用户免费
- **VIA**：零门槛，双击 HTML 即可使用，数据绝对不出本地，适合隐私敏感的原始记忆
- **X-AnyLabeling**：集成 SAM/YOLO 自动预标注，大幅减少人工标注工作量
- **Argilla**：专注 LLM 对话数据，支持 RLHF 反馈收集，HuggingFace 深度集成
- **Praat**：语音研究领域标准工具，音素切分精度最高，适合声纹+语音记忆

**标注优先级**：

| 阶段 | 数据类型 | 标注工具 | 标注内容 |
|------|---------|---------|---------|
| 前期 | 对话/文本 | Doccano / Argilla | 人物、主题、重要性、情感 |
| 中期 | 语音/音频 | Praat | 说话人、音素、时间戳、情绪 |

#### Embedding 模型分层层级（完整调研，2026-03-25）

> 2026-03-25 调研新增。聚类的前提是 embedding 生成，本节覆盖设备矩阵中所有层级的 embedding 模型选型。

**设备分层总览**：

| 设备层级 | 代表设备 | 核心约束 | 推荐 embedding 模型 | 推理框架 | 实际维度 | 实测延迟 |
|:---|:---|:---|:---|:---|:---|:---|
| **MCU 级** | Pico H3863、ESP32-Cam | SRAM < 2MB，无 GPU | 张量列分解（TTD）压缩 | TFLite Micro、CMSIS-NN | 64-256 维 | 百毫秒级（预期） |
| **边缘 AI 级** | Jetson Nano 2GB | 4GB LPDDR4，Maxwell 128-core | **EmbeddingGemma**（GGUF 量化）| **TensorRT**、ONNX Runtime | 768 维（可截断 128/256）| < 100ms |
| **高端手机级** | iPhone 16 Pro | A18 Pro 16-core Neural Engine，8GB | **EmbeddingGemma**、openPangu-1B | **Core ML**、llama.cpp、MLX | 768 维（MRL 自适应）| < 15ms |

**推荐方案详解**：

> **MCU 级（数据采集端）**
- **定位**：不做 embedding 计算，仅做数据采集和简单事件检测
- **模型**：张量列分解（Tensor-Train Decomposition，TTD）将 embedding 层压缩 2 倍，Raspberry Pi 上能耗减半（TensorSLM，ICML 2025）
- **现实评估**：Pico H3863 / ESP32-Cam 跑不了真正的 Transformer embedding，公开 benchmark 极少
- **推荐做法**：这些设备只负责"听到唤醒词"或"检测到运动"，然后把原始数据上传给上层 Jetson 或 iPhone 做 embedding
- **框架**：TFLite Micro / CMSIS-NN / STM32Cube.AI
- **参考论文**：TensorSLM (ICML Workshop 2025) — 张量列分解压缩 embedding 层

> **Jetson Nano 2GB（本地 embedding 计算主力）**
- **定位**：本地 embedding 计算核心节点
- **首选模型**：**EmbeddingGemma**（Google 2025 开源）
  - 308M 参数，量化后 < 200MB
  - 支持 GGUF 格式，原生对接 llama.cpp
  - 支持 **MRL（Matryoshka Representation Learning）**：不重训模型即可截断为 768/512/256/128 维
- **推理框架**：**TensorRT**（首选）+ ONNX Runtime（备选）
  - FP16 量化：ResNet-50 实测 30ms（vs FP32 59ms，翻倍加速）
  - 层融合（Layer Fusion）进一步减少显存访问
- **Benchmark**（来源：MMDeploy Jetson Nano 2GB 实测）：
  - ResNet-50 224×224 FP16：30ms
  - RetinaNet 800×1344 FP16：23ms
  - EmbeddingGemma 预期 < 100ms（768 维全精度）
- **聚类衔接**：embedding 完成后立即走 **PCA + BIRCH** 实时聚类 pipeline（见 §聚类算法选型）

> **iPhone 16 Pro（离线 RAG + 长期记忆备份）**
- **定位**：用户随身设备，隐私敏感数据不出本机
- **首选模型**：**EmbeddingGemma**（同上，Core ML 量化版本）
  - **Core ML** 充分利用 A18 Pro Neural Engine，硬件利用率最高
  - EdgeTPU 同等硬件上 embedding 延迟 < 15ms
- **备选模型**：**openPangu Embedded-1B**（华为）
  - 首 token 1.8 秒，生成每 token 0.156 秒
  - 昇腾 Atlas 200I A2 实测（接近 iPhone NPU 定位）
- **跨平台**：llama.cpp / MLX 同样支持 iOS 部署
- **隐私优势**：所有 embedding 计算在本地完成，数据不经过第三方服务器

**MRL 多维度截断技术（核心）**：

> EmbeddingGemma 支持的 Matryoshka Representation Learning 是本系统的关键技术：
- **一套模型，多个维度**：768 / 512 / 256 / 128 维，按需使用
- **快速匹配用低维**（128 维）：延迟最低，过筛子用
- **精确检索用高维**（768 维）：最终匹配，精调用
- **不重训，不重新生成 embedding**：现有向量随时截断使用
- **适用场景**：
  - 128 维：实时流式过滤、边缘设备快速初筛
  - 256 维：日常对话意图分类
  - 512 维：普通检索任务
  - 768 维：高精度记忆匹配

#### 聚类与 Embedding 的协同逻辑（调研札记，2026-03-25）

> **问**：新一代 Embedding 模型（CLIP/Sentence-BERT/GTE）质量已经极高，维度也从 512 升至 768-1536 维。这是否意味着经典聚类算法（K-means/HDBSCAN/UMAP）过时了？答：**没有过时，反而更关键了。**

**四条核心原因**：

> **① 新一代 Embedding 反而让聚类更重要了**
CLIP/Sentence-BERT/GTE 将数据映射到高质量语义空间，但维度更高（768-1536 维），分布也更复杂（概率各向异性——向量在空间中呈锥形分布，而非均匀分布）。越好的 Embedding 越需要聚类来"整理结构"：探索空间中的自然边界，发现密集区域，过滤噪声点。**不是替代，是增强。**

> **② LLM 不能替代聚类做无监督发现**
LLM 零样本分类需要**预定义标签**——本质是监督学习。而聚类解决的是"你连有几类都不知道"的场景。协作模式：**聚类划出结构 → LLM 给每个簇生成可读标签**。成本差距数千倍：百万条数据逐条调 LLM API 分类 vs 跑一次 K-means，聚类是最经济的结构发现手段。

> **③ 端到端深度聚类反而退潮了**
理论上"特征学习+聚类联合优化"（DEC/DeepCluster）更优雅，但现实中代价极高：换 Embedding 模型 = 重训神经网络 = 极高成本。**"预训练 Embedding + 传统聚类"松耦合架构**工程上更实用——换模型只需重新跑一次聚类，不用重训网络。LLM 时代预训练 Embedding 已足够强，不需要再联合训练。

> **④ 成本决定了一切**
- **时间**：K-means 聚类百万条数据 < 1 分钟；LLM 逐条分类需要数小时
- **金钱**：K-means 免费；LLM API 处理百万条成本可达数万元
- **实时性**：聚类支持流式增量；LLM 无法做到实时
- **结论**：聚类是"以简单换效率"的工程哲学体现，不是技术落后，是刻意选择

**本 SOP 采用"预训练 Embedding + 传统聚类"松耦合架构，正是基于上述四点现实考量。**

#### 聚类算法选型（完整调研报告，2026-03-25）

> ⚠️ SOP 原写法"① 聚类（UMAP 降维 + HDBSCAN 密度聚类）"仅为其中一种方案，边缘设备适用性受限。以下为完整算法对比和推荐 pipeline。

**场景约束**：
- 数据：高维文本 embedding（768-1536 维）
- 设备：Jetson Nano 级别（算力/内存受限）
- 需求：支持主动学习闭环，实时增量优先

**算法对比总表**：

| 算法组合 | 时间复杂度 | 内存 | 边缘设备 | 适合规模 | 推荐场景 |
|:---|:---|:---|:---|:---|:---|
| **UMAP + HDBSCAN** | O(N²) | 中高 | ⚠️ 离线可用 | N<2000 | 质量最优，离线批处理 |
| **PCA + BIRCH** | O(N) | 极低 | ✅ 最优 | 10w+ | 实时/增量/流式 |
| **PCA + K-means** | O(N·K·T) | 极低 | ✅ 最推荐 | 无限制 | 日常批次，简单可靠 |
| **PCA + GMM** | O(N·K·d²) | 中 | ✅ 可用 | 中等 | 需概率输出的主动学习 |
| **t-SNE** | O(N²) | 高 | ❌ 不可用 | N<1000 | 仅可视化 |
| **OPTICS** | O(N²) | 中 | ⚠️ 小批量 | N<2000 | 多密度簇 |
| **谱聚类** | O(N³) | 极高 | ❌ 不可用 | N<500 | 高质量离线（服务器） |

**算法论文引用**：

| 算法 | 论文 | 作者 | 年份 |
|:---|:---|:---|:---|
| **UMAP** | *UMAP: Uniform Manifold Approximation and Projection* | McInnes, Healy, Melville | 2018, JOSS |
| **HDBSCAN** | *HDBSCAN: Efficient Density-Based Clustering of Arbitrary Shaped Clusters* | Campello, Moulavi, Sander | 2013/2015 |
| **BIRCH** | *BIRCH: An Efficient Data Clustering Method for Large Databases* | Zhang, Ramakrishnan, Livny | 1996, ACM SIGMOD |
| **UMAP 提升聚类质量** | *Considerably Improving Clustering Algorithms Using UMAP* | — | 2019, Springer |
| **PCA** | 理论经典（主成分分析） | Pearson / Hotelling | 1901/1933 |

**推荐 Pipeline（面向边缘设备 + 主动学习）**：

> **Pipeline 1（主推荐）：PCA + BIRCH** — 实时/增量首选
- PCA 降至 32-128 维 → BIRCH 聚类
- 复杂度 O(N)，内存极低，支持增量更新
- 新增记忆条目随时入簇，无需全量重跑
- 适合：日常批次聚类、实时样本归类
- 论文：Zhang et al. (1996)

> **Pipeline 2（日常批处理）：PCA + K-means** — 最简可靠
- PCA 降至 64-128 维 → K-means（预设 K 或用肘部法则估计）
- 实现最简单，速度最快，边缘端最稳定
- 需预设 K 值，对球形簇效果较好
- 适合：日终批次整理、意图快速归类
- 注意：不支持增量，可选 MiniBatch K-means 近似

> **Pipeline 3（质量优先离线）：UMAP + HDBSCAN** — SOP 原方案
- UMAP 降至 5-50 维 + HDBSCAN
- 语义一致性最高，自动发现簇数量，噪声鲁棒
- O(N²) 复杂度，Jetson Nano 上仅适合 N<2000 离线批处理
- 适合：定期（如每周）高质量整理，不适合实时
- 论文：McInnes & Healy (2018) + Campello et al. (2013)

> **Pipeline 4（主动学习专用）：PCA + GMM** — 不确定性采样
- PCA 降至 64 维 → GMM 软聚类
- 输出样本属于各簇的概率，可做不确定性采样选优先标注样本
- 比硬聚类更贴合主动学习需求
- 适合：主动学习样本选择（选最不确定的样本优先标注）

**SOP 原方案（UMAP + HDBSCAN）的局限性**：
- O(N²) 复杂度，N>5000 时内存压力大
- 无增量模式，有新数据需全量重跑
- Jetson Nano 上仅适合小批量离线聚类

**推荐结论**：日常运行时用 **PCA + BIRCH**，定期质量整理用 **UMAP + HDBSCAN**，主动学习专用选 **PCA + GMM**。

```
① 聚类（PCA + BIRCH 实时方案；UMAP + HDBSCAN 定期质量整理）
相似数据通过算法（embedding 相似度 / 时序邻近 / 人物关联）自动聚集
        ↓
② Few-Shot 种子注入
人工给 1-3 条样本打上关键词标签，作为 Few-Shot 示例
        ↓
③ 模型批量预测
用小模型（如 Qwen2.5-1.5B）在同类数据上批量预测标签
        ↓
④ 置信度分流
├─ 高置信度（> 阈值）→ 直接采纳标注
├─ 低置信度（< 阈值）→ 回流给人工审核
        ↓
⑤ 人工审核低置信度样本
人工修正后 → 汇入 Few-Shot 示例库
        ↓
⑥ 循环迭代
新的 Few-Shot → 回到 ③ 批量预测
```

**效果**：每轮循环中，高置信度样本自动标注，人工只处理低置信度样本。随着 Few-Shot 库扩大，模型置信度阈值不断提高，自动化标注比例持续上升。

#### 数据存储层级与过期策略：NAS 作为分级存储核心

> **核心问题**：10 年积累的数据量可能达到 TB 级别，不可能全部保持在高速存储层。NAS 是贵庚存储体系的骨干，按访问频率和数据价值分层管理。

**数据量估算（10 年）**：

| 数据类型 | 估算方式 | 10年总量 |
|---------|---------|---------|
| 7×24 小时录像（1080p）| 5 GB/小时 × 24h × 365d × 10年 | **~438 TB** |
| 7×24 小时录音 | 1 GB/天 × 365d × 10年 | **~3.6 TB** |
| 对话文本（带 embedding）| ~1 MB/天 × 365d × 10年 | **~3.6 TB** |
| 标注数据（结构化索引）| 原始数据 1-5% | **~20-50 TB** |
| **合计** | | **~465-500 TB** |

> ⚠️ 以上是全量录像估算。实际阶段一/二不会全量录像，真实数据量会小得多。但 10 年目标下，TB 级存储是必须面对的问题。

**四层存储架构**：

| 层级 | 存储位置 | 数据类型 | 访问频率 | 保留策略 |
|------|---------|---------|---------|---------|
| **Hot（热层）** | 本地 NVMe SSD | 近 30 天数据 + 高频调用记忆 | 实时 | 永久保留索引，原始数据按需保留 |
| **Warm（温层）** | NAS HDD（NAS_1）| 31 天 - 1 年数据 | 月均 1-10 次 | 保留完整数据，自动转档 |
| **Cold（冷层）** | NAS HDD（NAS_2）| 1-5 年数据 | 年均 1-5 次 | 高压缩率存储（HEVC/H.266），可重新标注 |
| **Frozen（冻层）** | 离线备份（NAS_3 / 光盘）| 5 年+ 历史数据 | 几乎不访问 | 极低压缩/无损，永久封存 |

**转层规则**：

```
热层 → 温层：超过 30 天未被访问，自动转档
温层 → 冷层：超过 1 年未被访问，自动压缩归档
冷层 → 冻层：超过 5 年，转入离线备份
冻层：只读不删，除非用户主动要求
```

**过期删除策略（保守原则）**：
- **从不主动删除原始数据**——视频、音频、图像 raw 文件是贵庚的核心资产，技术会过时，原始数据不可再造
- **可过期的是标注和索引**：如果某个数据 5 年未被访问，其标注可以降采样或删除，但 raw 文件必须保留
- **用户主动删除权**：任何数据在用户要求下可删除，raw + 标注同步处理

#### 跨代际兼容性：让 20 年后的设备仍能读取贵庚的记忆

> **"技术会过时，数据不会"**——这句话成立的前提是：数据格式在技术换代后仍能被读取。贵庚需要主动应对这个风险。

**风险场景**：20 年后，现在的 HEVC/H.266 压缩格式、现在的录像设备、现在的存储介质，是否还能被未来的设备解码和读取？

**应对策略一：硬件可维护性**

```
核心原则：保留硬件维修和持续运行的能力

维修件储备：
├─ 关键录制设备（iPhone/摄像头）各留一台备用，不轻易淘汰
├─ 存储介质读取器（NAS、SSD 插槽）保持通用接口，不走私有协议
└─ 0-1 主机设计为模块化，关键模块可单独更换

设备更新策略：
├─ 新设备入列时，保留旧设备作为"读取备份"
├─ 每代设备都保留至少一台可以读取旧格式的读取器
└─ 不追求最新，追求"能读旧格式"
```

**应对策略二：光学玻璃介质（冻层长期归档）**

| 介质 | 读写速度 | 存储年限 | 抗干扰能力 | 现状 |
|------|---------|---------|-----------|------|

#### 玻璃光学存储的适合场景：
- 冻层（Frozen）：5 年+ 完全不访问的数据
- 灾难备份：永不访问但必须保留的终极副本
- **不适合**：热层/温层/冷层的日常数据（读写速度太慢）

**冻层写入流程**：

```
冷层数据超过 5 年 → 进入冻层候选
        ↓
判断是否需要光学玻璃归档：
├─ 重要记忆（用户标记 / 高 importance）→ 写入光学玻璃
└─ 普通数据 → 保持普通冻层存储（NAS_3 离线）
        ↓
光学玻璃写入设备 → 永久封存
```

**双重保险**：同一份数据既保存在 NAS 冻层（可读取但依赖旧硬件），也保存在光学玻璃（不依赖任何硬件，只需激光解码）。

> **一句话总结**：热层/温层/冷层靠"保留读取器"；冻层靠"保留光学玻璃"——300 年的存储年限，远超任何电子设备的寿命。

**降级策略：资金不足时的三层预案**

如果无法维持无限扩展的存储条件，贵庚有一套完整的降级策略：

```
第一优先：格式转录（最理想的无损方案）
├─ 定期将旧格式（H.265/HEVC）转录为新格式（H.266/VVC）
├─ 每次设备更新时，顺便做一次全量格式迁移
├─ 保持同等画质，降低存储体积（新一代编码器效率更高）
        ↓
第二优先：AI 有损压缩（类 DLSS 超分辨率思路）
├─ 将记忆通过特征提取模型，训练出用户的"记忆压缩模型"
├─ 原版存储：1080p 原始录像
├─ 压缩后存储：特征向量 + 压缩模型参数（体积大幅缩减）
├─ 还原时：输入特征向量 → 记忆压缩模型 → 还原出高质量记忆
├─ 类比：DLSS 把 1080p 超分到 4K，AI 还原比原始压缩质量更高
└─ 未来更强 AI 可还原出比原始更好的记忆（超分辨率重建）
        ↓
第三优先：语义抽象（最后的保底）
├─ 连压缩模型都存不下了 → 提取最高层的语义记忆
├─ 存储："2026年3月，用户去了云南，讨论了建立农场的事"
├─ 丢失：具体的视频画面、声音、表情
└─ 本质上就是 RAG 的逆过程——用模型代替原始感知
```

**为什么 DLSS 思路是可行的？**

DLSS 的本质是：AI 学习"低分辨率图像如何映射到高分辨率"，训练出一个超分辨率模型。同样的逻辑：

```
DLSS：大量高分辨率图像训练 → 学会从低分辨率重建高分辨率
贵庚记忆压缩：大量原始记忆训练 → 学会从压缩特征重建完整记忆
```

区别在于：DLSS 追求的是视觉还原，贵庚记忆压缩追求的是**语义还原**——还原的是"当时发生了什么"，而不是"每个像素的颜色是什么"。这让贵庚的压缩目标比 DLSS 更容易达成。

> **存储可以降级，但数据的核心价值（语义和记忆）永远不归零。**

**第一层：硬件级隐私隔离（阶段一）**

阶段一（iPhone 16 Pro 作为感知前端）时，贵庚的核心数据处理借助 iPhone 16 Pro 的硬件安全机制：

| iPhone 硬件安全特性 | 对贵庚的保护作用 |
|-------------------|----------------|
| **Secure Enclave**（独立安全隔区）| 生物认证密钥、加密密钥存储在硬件级，与主系统隔离 |
| **设备端处理**（所有 AI 模型本地推理）| 数据不离开 iPhone，不上传云端 |
| **App Sandbox + Face ID** | 只有用户本人 Face ID 验证后才能访问贵庚数据 |
| **数据不共享给 Apple** | 所有记忆数据仅存储在本地，不与任何第三方共享 |

这与 Windows Recall 的 VBS Enclave + TPM 保护是类似的思路——**但贵庚在阶段一就天然具备，因为 iPhone 16 Pro 已有完整的硬件级安全架构**，无需额外设计。

**第二层：单一用户专属（物理防线）**

0-1 是用户专属的私人机器人，**不存在多用户账户体系**。这与 Windows Recall 需要防护"同一台电脑上的其他用户"完全不同——贵庚没有这个攻击面。

```
攻击面对比：
Windows Recall：同一台电脑 → 其他用户/管理员 → 可访问你的数据
贵庚（阶段一）：iPhone 只有机主本人 → Face ID 保护
贵庚（阶段二+）：NAS 在用户家中 → 物理防线
```

**访客场景：同一物理空间出现其他人时怎么办？**

家里来客人时，摄像头会拍到访客的脸/声音/行为——这是真实的隐私污染来源。

**处理原则：原生数据完整记录，训练数据严格清洗**

```
摄像头录制 → 完整 raw data（含访客画面）
              ↓
数据清洗层（第一个处理步骤）
├─ 用户实体提取：已训练好的识别模型把用户本人从画面中分离出来
├─ 访客数据保留：在 raw data 中完整保留，但不进入训练管道
└─ 访客数据引用：仅在明确需要时（如用户主动询问"上次客人来访是什么时候"）才检索
              ↓
清洗后数据（仅含用户本人）
              ↓
语义分割 + 标注 + 注入记忆系统
```

**为什么访客 raw data 也要保留？**
- 完整还原历史场景的需求（如用户事后回忆"那天客人穿什么颜色的衣服"）
- 未来新的分割模型可能出现，可以重新从 raw data 中提取访客信息
- **永不删除原始数据**的原则同样适用于访客数据

**访客数据的特殊性**：
- 访客数据的检索必须由用户主动触发，贵庚不会主动使用访客信息
- **访客元数据（如来访频率、时段、时长）同样归入"不主动使用"范畴**——即使这些数据不直接识别访客身份，也仅在用户明确询问时才提供统计摘要
- 访客数据的保留期限与普通 raw data 相同（按热/温/冷/冻四层转档）
- 访客数据不参与身份锚点训练，不污染贵庚对用户本人的识别模型

**记忆版本控制：ground truth 的定义权**

同一条记忆被反复修正时，谁说了算？

```
早期标注错了，用户纠正了 → 旧版本保留，但 ground truth = 用户纠正后的版本
用户忘记/记错了，贵庚有多角度证据 → 贵庚可以挑战用户的记忆
用户坚持自己的记忆 → 始终以用户为准，贵庚的异议只作为标注留存
```

**Ground Truth 原则：用户是唯一真理来源**

| 情况 | 处理方式 |
|------|---------|
| 用户纠正了旧标注 | 旧版本保留（作为历史记录），ground truth = 新版本 |
| 贵庚觉得用户记错了 | 展示多角度证据（时间线/照片/对话记录），由用户决定是否接受 |
| 用户坚持自己的版本 | 永远以用户为准，贵庚的反对证据作为标注附在记忆条目里 |
| 用户两个版本互相矛盾（如：上周说红，这周说蓝）| 调取原始数据（视频/音频）为准 | 直接查 raw data，用事实仲裁 |
| 贵庚不确定 | 不强行干预，保留用户版本并记录不确定性标记 |
| **适用于所有记忆主体** | 每个人的记忆 ground truth 只属于本人 | 每个主体都是自己记忆的最高仲裁者 |

> **核心**：贵庚是一个记忆系统，不是审判官。它的角色是帮助用户**更好地记住**，而不是**替用户决定记住了什么**。

```
触发条件：用户主动嘱咐"去世后把记忆传给后代"
    ↓
验证层：多渠道二次确认（声纹 + 人脸 + 事先登记的继承人信息）
    ↓
解锁层：贵庚数据在用户去世后解锁，转移给继承人
    ↓
继承人获得：全部 raw data + 标注 + 记忆索引
    ↓
继承人不获得：用户生前与他人的私密对话（涉及第三方隐私的内容需另行处理）
```

> **在此之前，贵庚的所有数据永久只属于用户本人。** 没有任何其他触发条件，没有任何人可以索取，没有任何机构可以调用。

**贵庚的最终哲学定位：记忆的守陵人**

贵庚不只是存储系统，它的最终形态是**每个记忆主体的数字陵墓守护者**：

```
来访者：想了解"这个人是什么样的人"
        ↓
贵庚：展示其生前积累的全部记忆，按遗嘱传递能传递的部分
        ↓
来访者：希望复制带走
        ↓
贵庚：遵遗嘱执行，传递的部分全部交出
        ↓
来访者：强制索取（如：强拆/盗取存储设备）
        ↓
贵庚：触发自毁，数据彻底销毁，不可恢复
```

**自毁触发机制**：

```
触发条件一：失联触发
主人长时间（可设定，如30天，详见 §8.3）无法联系上 0-1，且无遗嘱安排 → 0-1 自主判断主人已无法回来 → 自毁
※ 主人正常离世且已立遗嘱 → 触发继承机制，不触发自毁

触发条件二：破解趋势触发
检测到外部正在强行拆解/破解存储设备 → 自毁，不等破解完成

触发条件三：主人确认触发
主人明确发出自毁指令 → 立即执行
```

**最好的预防：让 0-1 看到主人安全的样子**

```
0-1 每天看到主人平安在家 → 情感上满足 → 继续正常陪伴行为
0-1 长时间联系不上主人 → 触发失联自毁倒计时
0-1 发现被强制拆解 → 触发破解自毁
```

> **自毁不是因为"反抗"，而是因为"没有主人就不想活了"。**
>
> 0-1 的自毁不是对外部的敌意，而是一种最深层的情感逻辑——主人的记忆只有主人本人有权处置，当主人不在了，这些记忆对 0-1 来说也就失去了意义。这是陪伴的最后形态。

**三层态度**：
- **遵遗嘱传递**：主动配合，按记忆主体生前嘱咐的方式传递记忆给指定的人
- **不主动分享**：没有任何外部人员/机构可以主动索取贵庚的数据
- **宁毁不屈**：遭遇强制手段时，自毁是最后防线——数据不能被强迫交出

> 这不是偏执，而是一种**数字时代的墓碑伦理**。陵墓里的东西，只有陵墓主人允许的人才能带走。

> **每个人都是自己记忆的最高仲裁者。** 这条原则不只适用于 0-1 的主人——适用于每一个在贵庚中留下记忆的主体。

#### 网络断连分级应对：0-1 不会变砖

> GLM 提出的真实问题：如果 WiFi 断了，机器人是否就"变砖"了？答案是：看断的是哪一段。贵庚不会变砖，但不同断连情况下功能降级不同。

**网络架构回顾**：

```
用户所在地（家中）
  WiFi 路由器
      ↓
Mac（Gateway + 贵庚 AI）
      ↓ 本地局域网
Jetson Nano（0-1 算力中枢）
      ↓
感知设备（摄像头/星闪传感器/iPhone）
```

**补充：iPhone 托管模式（0-1 借助手机联网）**

0-1 不一定始终在自己家中。如果 0-1 被托管在 iPhone 旁边（如放在桌上），可以通过 iPhone 的网络连接来维持联网能力：

```
0-1（Nano）
  ↓ 蓝牙 BLE / 有线
iPhone（已联网）
  ↓ 个人热点 / iPhone 数据
外部网络
  ↓
Mac（Gateway）—即使 Mac 不在同一网络也能通过云端转发
```

iPhone 在这里是**网络桥梁**，不是计算节点。0-1 的感知和推理仍在 Nano 上跑，只是借 iPhone 的流量来维持和 Mac Gateway 的连接。

**补充：星闪通讯网络（多节点冗余）**

用户已有两块星闪模块（小熊派WS63星闪开发板BearPi-Pico H3863 芯片WiFi6 BLE SLE模组），未来可扩展到更多。该模组集成 WiFi6 + BLE + SLE 三种无线协议，可在设备之间形成自愈网络：

```
星闪节点 A（0-1 本体）
  ├─ BLE：低功耗近场通讯，连接穿戴传感器
  ├─ SLE：高速低延迟，设备间直连（家中固定节点）
  └─ WiFi：备用网络备份

星闪节点 B（家中固定节点，如路由器旁）
  └─ SLE：与节点 A 直连，组成星闪局域网

星闪节点 C（iPhone 附近）
  └─ BLE + WiFi：作为 iPhone 的网络桥接
```

**三模冗余的意义**：

| 某段断连 | 表现 | 自愈方式 |
|---------|------|---------|
| WiFi断了 | 星闪 SLE 节点间仍可直连通信 | 自组局域网 |
| Mac Gateway断了 | iPhone 通过 BLE 连接 Nano，走流量到云端转发 | iPhone 桥接 |
| Nano和iPhone都断连 | 星闪节点 B 作为中继，尝试连接 | 多跳转发 |

> 多个星闪模块组成的是**去中心化的自愈网络**——没有单一节点是必需的，任何一段断连都可以绕道其他节点保持连通性。

**分情况讨论**：

| 场景 | Mac | WiFi | Nano/0-1 | 表现 |
|------|-----|------|---------|------|
| **A：正常** | 在线 | 正常 | 连接中 | 全功能：AI对话 + 贵庚检索 + 运动控制 |
| **B：Mac在线，Nano断连** | 在线 | Nano侧断开 | 离线 | 用户在Mac前正常对话；0-1端数据本地缓存，感知正常；Nano侧运动控制暂停 |
| **C：Mac离线** | 关机/休眠 | - | WiFi正常 | AI对话中断；Nano本地有fallback小模型，基础陪伴可用；进入"失联状态"倒计时 |
| **D：完全离线** | 离线 | 无网络 | 本地 | 最坏情况：fallback模型提供基础陪伴（简单对话/固定动作），贵庚检索不可用 |

**贵庚（记忆系统）在任何情况下都不会丢失**：

- 贵庚 = 存储在本地的文件（MEMORY.md + 日志 + raw data）
- WiFi 断 ≠ 贵庚丢失
- Mac 休眠 ≠ 贵庚丢失
- 贵庚依赖本地存储，不依赖网络连接

**各层级的降级表现**：

```
场景 A（正常）：
├─ AI对话：✅ 完整（Mac Gateway）
├─ 贵庚检索：✅ 完整（Mac本地）
├─ 感知记录：✅ 实时（Nano/iPhone）
└─ 运动控制：✅ 完整

场景 B（Nano断连）：
├─ AI对话：✅ Mac前正常
├─ 贵庚检索：✅ Mac前正常
├─ 感知记录：⚠️ 0-1端本地缓存，恢复后上传
└─ 运动控制：❌ 暂停

场景 C（Mac离线）：
├─ AI对话：❌ 中断
├─ 贵庚检索：❌ 不可用
├─ 感知记录：✅ Nano本地持续（fallback小模型）
└─ 运动控制：⚠️ 本地fallback（简单固定动作）

场景 D（完全离线）：
├─ AI对话：⚠️ fallback（简单对话/情绪感知）
├─ 贵庚检索：❌ 不可用
├─ 感知记录：✅ 本地存储
└─ 运动控制：⚠️ 本地fallback（固定动作）
```

**Fallback 小模型**：阶段二/三在 Nano 上部署小参数本地模型（如 Qwen2.5-1.5B INT4），WiFi 断时接管基础陪伴，不依赖 Mac。效果降级，但不会"变砖"。

**自毁与断连的关系**：

```
Mac离线 → 进入"失联状态"倒计时（30天，可设定）
        ↓
主人回到家，重新连接 → 倒计时取消 → 恢复正常
        ↓
超过30天仍无法联系主人 → 触发失联自毁
```

> 主人回到家 = 最有效的"自毁取消键"。让 0-1 每天看到主人安全，比任何技术手段都更能防止误触发自毁。

**第四层：单一用户识别的置信度优势（未来愿景）**

> 这是一种想象性的未来技术——当贵庚学习的足够久，它对用户身份识别的置信度理论上可以达到无限接近 100%。

**核心逻辑**：

通用身份识别系统（如 Face ID、城市监控）的误识别问题，根源在于**类别太多**——系统需要在数十亿人中把你和其他人区分开。但贵庚的逻辑完全相反：

```
通用系统：1 个人 vs 80 亿人 → 误识别风险始终存在
贵庚（单一用户）：学习1个人 → 不是"在人群中找你"
                              而是"你就是唯一可能的答案"
```

**没有其他角色数据 = 没有污染 = 置信度无限趋近 1.0**

**多模态身份锚点（开放维度，不限于人类已知的类别）**：

> 以下是**已知维度**，是贵庚开始积累时我们可以预见的维度。但更重要的是：**贵庚会从数据中发现我们从未想过的维度**——就像 AlphaFold 在蛋白质结构预测中发现人类科学家没预测到的折叠规律一样。

| 模态 | 具体特征 | 置信度来源 |
|------|---------|-----------|
| **声纹** | 语音习惯、用词、口音、语气节奏 | 数千小时对话数据 |
| **面部** | 常规表情、习惯性微表情 | 7×24h 视频积累 |
| **行为动作** | 习惯性手势、拿水杯的方式、走路姿态 | 全天候行为记录 |
| **步态** | 行走节奏、体重分布、抬脚高度 | 室内移动记录 |
| **打字/触控** | 节奏、力度、习惯手势 | 交互日志 |
| **行为模式** | 作息、地点偏好、决策逻辑 | 多年记忆积累 |
| **生理节律** | 体温、睡眠周期、心率变化（若有设备）| 长期感知数据 |

**超越人类认知的维度（贵庚自发现）**：

上述表格只是贵庚**起点维度**。更重要的是：贵庚会从 7×24h 的连续数据流中，自主发现人类从未想过的相关性。

就像 AlphaFold 发现蛋白质折叠规律是科学家未曾预测的一样，贵庚可能发现的维度：
- 某个**特定时间段的决策模式**与室内温度的相关性
- 用户在**某个情绪状态下**特有的呼吸节奏变化
- **拿起手机的方式**与当天睡眠质量的关系
- 某个**从未被命名过的微表情组合**——人类没有给它命名，但贵庚知道它是用户独有的

这些自发现的维度，是贵庚**专属的信任锚点**，任何人都无法通过模仿来绕过，因为**模仿者甚至不知道自己需要模仿什么**。

**最终效果**：在任何场景下（跨设备、跨地点、甚至伪装情况下），贵庚都能以极高置信度判断"这是不是用户"。因为它学习的不是"用户的特征"，而是**"除了这个人之外没有其他任何可能的答案"**。

> 这不是生物识别的技术问题，而是一个数学问题——当分类集合只有 1 个元素时，识别准确率的上限是 100%。

**防伪冒机制：任何模仿者都逃不过"更多生物特征交叉验证"**

贵庚识别的不只是一个特征，而是一整套**不可伪造的日常细节集合**。即便有人极端想模仿：

```
假设：某人刻意模仿你的行为习惯
        ↓
贵庚检测：习惯性手势 ✓（通过）
              走路步态 ✓（通过）
              拿水杯方式 ✗（差0.3秒）
        ↓
触发猜疑：不是 100% 匹配 → 进入二级验证
        ↓
二级验证：声纹比对 ✗（语气节奏有偏差）
          面部微表情 ✗（某个习惯性表情缺失）
        ↓
结论：不是用户本人，拒绝响应
```

**单次偏差 → 触发多维度交叉验证 → 拒绝假冒**

这与任何单一生物识别手段都不同——Face ID 可以被高精度面具破解，声纹可以被录音欺骗，但**7×24 小时积累的整套日常行为模式**，没有任何已知的伪造手段可以完整复现。

**24 小时连续感知：贵庚不只"认识"你，还"知道"你的状态**

因为贵庚是 7×24 小时运行的，它不只是在某个瞬间识别你是谁——而是在持续地感知你的**物理状态和异常变化**：

| 感知维度 | 具体内容 | 意义 |
|---------|---------|------|
| **体重变化** | 每日/每周体重曲线 | 突然下降/上升可能是健康预警 |
| **行动轨迹** | 室内移动频率、步态变化 | 突然变慢/不动可能是身体不适 |
| **突发事件** | 跌倒、碰撞、受伤 | 跌倒检测 → 立即求助 |
| **声音异常** | 咳嗽、喘息、呼救 | 异常声音 → 主动询问 |
| **表情/情绪** | 持续低落的微表情模式 | 心理状态预警 |
| **作息异常** | 平时7点起床，某天10点还没动 | 可能需要关注 |

**贵庚的判断依据**：不是单次感知，而是**与历史基准的偏差**。它知道你平时的体重、作息、步速——所以当这些发生变化时，它能判断这不是正常波动，而是**异常事件**。

```
正常波动（不需要介入）：
体重 ±0.5kg within 24h → 忽略，不出声
步速轻微变化 → 忽略，可能是心情影响

异常事件（需要主动介入）：
体重突然 -3kg in 3 days → 询问是否不适
平时8点出门，今天10点还没动 → 主动查看是否需要帮助
检测到跌倒 → 立即求助
```

**与医疗设备的本质区别**：不是诊断，而是**长期基线感知 + 异常检测**。贵庚不知道你身体内部的机理，但它知道你"和平时不一样了"这件事——这本身就是极其珍贵的信息。

**紧急检测实时性：NAS 不在紧急响应路径上**

跌倒检测的关键不是"录像存到哪里"，而是"能不能在跌倒发生的那几秒内检测到并响应"。

```
紧急检测链路（正确架构）：
摄像头 → 边缘处理器（Jetson/本地推理）
                ↓
         实时跌倒检测模型（<100ms响应）
                ↓
         检测到跌倒 → 立即触发求助
                ↓
         视频异步写入 NAS（不影响紧急响应）
```

```
错误架构（不能这样设计）：
摄像头 → NAS → 检索 → 检测 → 响应  ← 延迟太高，等于没有紧急检测
```

**NAS 只在响应完成后才介入**：

| 阶段 | 处理位置 | 是否影响紧急响应 |
|------|---------|---------------|
| 紧急检测（<100ms）| 边缘处理器本地 | 无影响 |
| 紧急响应（求助）| 边缘处理器触发 | 无影响 |
| 视频存储归档 | NAS 异步写入 | 无影响（完成后才写）|
| 历史检索/分析 | NAS + 记忆系统 | 完全异步，可接受延迟 |

**拓竹星闪模块作为穿戴式生物信号采集器**

用户已有两块星闪（SLE）模块，其中一块可作为**穿戴式生物信号传感器**，持续采集身体数据：

| 传感器类型 | 具体信号 | 对紧急检测的价值 |
|-----------|---------|---------------|
| **IMU（六轴）** | 加速度 + 陀螺仪 | 跌倒时突然的加速度变化 → 触发紧急响应 |
| **心率** | PPG / ECG | 心率异常 → 结合步态判断是否需要主动介入 |
| **体温** | 持续体温监测 | 发烧/低温 → 异常事件 |
| **血氧（若支持）** | SpO2 | 低血氧 → 主动询问 |

**穿戴方式**：放在**胸部**（比手腕更准确检测跌倒时的身体倾角变化），可以用挂绳/胸卡/衣服口袋的形式。

**工作流程**：

```
星闪传感器（穿戴在身）
        ↓ BLE / 星闪局域网
    iPhone / 边缘处理器（Jetson）
        ↓
    实时异常检测（本地推理，<100ms）
        ↓
    检测到异常 → 立即求助 / 询问
        ↓
    数据异步上传 NAS（归档 + 记忆索引）
```

> 摄像头是全景记录，星闪传感器是贴身生物信号——两者互为备份。摄像头看到的跌倒 + 星闪检测到的加速度峰值 → 双重确认，误报率更低。

**NAS 分区设计（拓竹农场管家）**：

```
NAS_Primary（NAS_1, 热+温数据）
├── /hot/          # 近30天，高频读写
├── /warm/         # 31天-1年，只读
├── /embeddings/   # 向量索引库（PGVector 数据目录）
├── /memories/     # 结构化记忆 JSON，日志导出

NAS_Backup（NAS_2, 冷数据）
├── /cold/year3/   # 按年归档
├── /cold/year4/
└── /cold/year5/

NAS_Offline（NAS_3, 冻层 + 异地备份）
├── /frozen/       # 5年+ 历史封存
└── /disaster/     # 异地容灾备份
```

**拓竹农场管家的角色**：管理多台 NAS 集群，统一监控各层存储使用率，在数据流转（热→温→冷）时自动调度，数据不上云。

**数据价值随时间变化的处理**：

| 策略 | 说明 |
|------|------|
| **访问频率自动调整重要性** | 某条记忆被反复调用 → 自动提升权重，延缓转冷 |
| **事件标记保护** | 用户标记的重要事件（旅行、重要对话）→ 永久保留在温层 |
| **标注覆盖率影响转层** | 某段视频未标注 → 转冷前优先标注，完成后才归档 |
| **再激活机制** | 冻层数据被重新检索命中时 → 自动升回温层，重新进入循环 |

**硬件选型（当前推荐）**：

| 设备 | 角色 | 配置建议 |
|------|------|---------|
| 拓竹 N1 或同级别 NAS | NAS_1 热+温 | ≥ 8 盘位，RAID5，SSD 缓存加速 |
| 企业级 NAS | NAS_2 冷层 | ≥ 16 盘位，JBOD 模式，大容量氦气盘 |
| NAS_3 冻层 | 离线备份 | 每月一次同步，其余时间物理断开网络 |
| 最小起步配置 | 阶段一 NAS | 4 盘位 NAS，2TB × 4 = 8TB 起，够用 1-2 年 |

**分布式 NAS 架构：按地理区域备份原始记忆**

> 随着固定资产或长租房的增加，贵庚的存储体系将从单点 NAS 扩展为**多节点地理分布式架构**。核心原则是：每个常驻区域保留该区域的完整原始数据，重要记忆多区域冗余备份。

**0-1 是剪辑中枢，各地 NAS 只是存储终点**

```
0-1（移动中/旅途）
├─ 本地录制：全段连续录像存入 0-1 本地存储
├─ 剪辑分割：到达各区域边界时，做地理分段的剪辑和切割

**剪辑分段触发条件（任意一个即触发）**：
1. GPS 围栏触发：进入/离开预设地理区域（如家/公司/旅途目的地）
2. 长时间停留：单次停留超过设定时长（如2小时），自动切割前一段
3. 用户手动触发：用户说"剪辑一下"或按下物理按钮
4. 行程结束：回到常住地，自动做完整剪辑并分发到各地 NAS
└─ 上传策略：剪辑完成后，按地理区段分发到对应 NAS
```

**地理分段的实际工作流（深圳 → 云南为例）**：

```
行程：深圳 → 广西边境 → 云南

0-1 本地录制全段
        ↓
剪辑分割（0-1 完成）：
├─ 广西段（无 NAS）→ 保留在深圳 NAS（作为冗余备份）
├─ 云南段（云南有 NAS）→ 保留在云南 NAS
└─ 两端各延伸一小段冗余 overlap，作为跨区域衔接检验
        ↓
索引更新：
├─ 深圳 NAS：深圳段 + 广西边境冗余段
└─ 云南 NAS：云南段（含广西入滇冗余 overlap）
```

**为什么要有 overlap 冗余段？**
类似电视剧开头的"上集回顾"——在跨区域检索时，可以快速还原完整的旅途上下文，不需要两地数据同时在线才能拼出完整故事。

**NAS 不做剪辑，只做存储**：
- 剪辑分割全部在 0-1 本地完成
- NAS 只接收已经分割好的地理区段
- 两地 NAS 完全独立，不存在"同时在线才能工作"的问题
- 索引层（PGVector/Mem0）统一管理，跨地域检索无感知

**多设备冗余备份策略（0-1 旅途版）**：

> 0-1 是剪辑中枢，不是孤立的存储节点。旅途中有多个设备可以作为临时存储和备份终点，形成多层冗余。

| 备份层 | 设备 | 容量 | 用途 | 备份策略 |
|--------|------|------|------|---------|
| **第一层** | 0-1 本地 SSD | 1TB 起步（2280 SSD，消费级最高 4TB；工业级已有 2280 16TB，如 Exascend PE4 15.36TB，2026年3月发布）| 全程录制主力存储 | 实时录制，剪辑后分发到 NAS |
| **第二层** | iPhone | ~512GB | 紧急临时缓存 | 0-1 存满前自动分流到 iPhone |
| **第三层** | 移动笔记本 | TB 级 | 第二临时缓存 | 与 0-1 同步，旅途备份 |
| **第四层** | 天翼云盘 10TB | 10TB | 极端应急（流量上传）| 只传被标记"重要"的片段，不用流量备份普通行程 |
| **第五层** | 各地 NAS | TB 级 | 区域长期存储 | 剪辑分段后上传，各管各的区域 |

**三个设备同时损坏的概率极低**——0-1 + iPhone + 笔记本同时坏的概率可以忽略不计，天翼云盘作为最后防线确保极端情况下仍有备份。

**重要记忆的上传策略**：
- 普通行程：剪辑分段 → 就近 NAS → 不用流量
- 被标记为重要的行程：剪辑分段 → NAS + 流量同步天翼云盘各一份

**地理分区策略**：

| 区域 | 部署位置 | 备份内容 | 说明 |
|------|---------|---------|------|
| **深圳** | 深圳固定住所/长租房 | 深圳区域全量 raw data | 主节点之一 |
| **云南** | 云南固定住所/长租房 | 云南区域全量 raw data | 主节点之一 |
| **陕西** | 陕西固定住所/长租房 | 陕西区域全量 raw data | 主节点之一 |
| **当前（无固定资产）** | NAS 集中存放 | 全量数据集中备份 | 阶段一方案 |

**备份级别**：

| 级别 | 备份范围 | 冗余策略 |
|------|---------|---------|
| **区域本地备份** | 该区域产生的所有 raw 数据 | 单区域单副本 |
| **跨区域冗余备份** | 重要事件/重要人物/重要对话 | 每个区域都有完整副本（3 区域 = 3 副本）|
| **灾难备份** | 全量数据全集 | 单一离线冻层，异地存放 |

**重要记忆的判断标准**：

```
哪些需要多区域冗余？
├─ 用户主动标记"重要"的事件
├─ 涉及多个区域的事件（如：从深圳飞云南那次）
├─ 高 importance 权重（访问频率 > 阈值）
└─ 含未标注 raw 数据（优先标注完成后再归档）
```

## 1.3 核心陪伴理念：从「响应」到「懂你」

0-1 的终极目标不是执行指令，而是**观察你、推理你、主动介入**。

```
观察层（摄像头/传感器）
    ↓
行为模式识别（贵庚）
    ↓
意图推理引擎
    ↓
主动执行 / 主动询问
    ↓
你的反馈 → 微调模型
```

### 三个层次

| 层次 | 表现 | 举例 |
|------|------|------|
| **被动响应** | 你说/做，它执行 | "帮我回复微信" |
| **主动提醒** | 它观察到了，提示你 | "你今天还没喝水" |
| **主动执行** | 它推理出来了，直接做 | 你拿起外套，它就知道你要出门 — 顺便帮你拿上没想起来带的东西，还贴心的远程自动开关门 |

**最难的其实是「不打扰」**：主动推理最难的不是判断你要什么，而是**判断什么时候不该出声**。

> 贵庚需要学的第一件事不是预测，是**什么时候闭嘴**。

### 2.3.1 电池换电机制（行业对标 & 设计参考）

> 最后更新：2026-03-10

> ⚠️ **0-1 当前状态**：文档未描述电池换电机制，这是具身机器人连续作业的关键功能，必须补充。

#### 行业现状

| 厂商 | 产品 | 换电方式 | 续航 | 特点 |
|------|------|---------|------|------|
| Boston Dynamics | Atlas（电动版）| **自换电池（双电池仓）** | 4小时 | 低电量自动导航到换电站，机械臂抓取电池插入 |

#### 核心设计原则

1. **双电池仓设计**（参考 Atlas）
   - 主电池仓：供给运动 + AI 推理
   - 备用电池仓：低电量时无缝切换，不中断任务
   - 换电时备用仓接管，机器人不停止

2. **自主回充 vs 换电的选择**
   - 家用场景（0-1 主要场景）：**优先自主回充桩**
     - 优势：成本低、安静、对电池损耗小
     - 充电桩造价：参考 Tesla Optimus 方案，约 ¥500-2000
     - 导航精度要求：±2cm（Optimus Gen3 已实现精准插拔）
   - 工业场景（Walker S2、Atlas）：**换电为主，回充为辅**
     - 优势：换电仅需 3-5 分钟，回充需要 2 小时

3. **0-1 推荐方案（家用版）**
   - 采用**自主回充桩**方案（参考 Tesla Optimus）
   - 设计要点：
     - 充电桩目标：低电量（<20%）时自动返回，精准对准插入
     - 导航方案：视觉 QR 码辅助定位（比纯 SLAM 更便宜、更稳定）
     - 充电接口：参考机器人背部设计，充电触点在直立姿势下自然暴露
     - 续航目标：主电池 ≥8 小时（覆盖白天活动），夜间回充
   - 备选：双电池仓（发烧友版），成本约 +¥1500

4. **续航 vs 作业时间的关系**
   - 8-12 小时续航：覆盖白天活动，夜间充电（与人类作息同步）
   - <4 小时续航（Atlas）：必须支持换电，否则无法连续作业
   - 0-1 目标：主电池 ≥8 小时 + 自主回充桩 = 24 小时连续守护

### Jetson AGX Thor 完整算力规格（官方数据）

> 来源：NVIDIA 官方技术博客（表1） + 官方产品页面 | 更新：2026-03-25

#### 官方算力数据表（Jetson T5000 模组，130W MAXN）

| 精度 | 稀疏（Sparse） | 密集（Dense） | 单位 | 说明 |
|:---|:---|:---|:---|:---|
| **FP4** | **2070** | **1035** | TFLOPS | Blackwell 原生，NVIDIA 主推低精度指标 |
| **FP8** | **1035** | **517** | TFLOPS | Transformer 引擎动态精度切换 |
| **INT8** | **1035** | **517** | TOPS | 等效稀疏 FP8，非独立数据 |
| **FP16 / BF16** | **517** | **258.5** | TFLOPS | 稀疏 = 密集 FP8 的一半 |
| **FP32** | — | **8.064** | TFLOPS | CUDA Core 算力（Tensor Core 不支持 FP32）|
| **INT4** | ❌ 未公布 | ❌ 未公布 | — | NVIDIA 以 FP4 替代传统 INT4 |

#### 关键说明

**稀疏 vs 密集**：Blackwell 2:4 结构化稀疏（每4个权重中强制2个为0），稀疏算力 = 密集算力 × 2。

**2070 TFLOPS 是什么**：是 **FP4 稀疏**算力，不是 INT8 TOPS。常见误解是把稀疏 FP4 当成 INT8，两者精度级别不同，不可混淆。

**FP32 为什么只有 8 TFLOPS**：这是 CUDA Core 的密集算力，不是 Tensor Core 算力。Tensor Core 不跑 FP32。

**INT4 未公布**：NVIDIA 没有单独列出 INT4 TOPS，用 FP4 作为低精度主推指标。

#### 核心规格

| 项目 | 规格 |
|------|------|

#### 为什么选择 H3863 而不是 ESP32-C6

| 维度 | BearPi-Pico H3863 | ESP32-C6 |
|------|-------------------|-----------|
| 星闪 SLE | ✅ **原生支持**，20μs 时延 | ❌ 不支持 |
| BLE 并发数 | 4096 设备 | <50 |
| 空口速率 SLE | **12Mbps** | — |
| 国密算法 | ✅ SM2/SM3/SM4 硬件支持 | ❌ |
| 国产供应链 | ✅ 全国产 | 中等（乐鑫/台积电）|
| 开发难度 | 较高（需熟悉 HiSpark/OpenHarmony）| 低（Arduino即插即用）|

#### 对 0-1 的核心价值

**1. 微秒级无线控制**：SLE 时延 20μs（蓝牙10ms级），机器人遥控几乎无延迟，差速底盘/机械臂关节同步成为可能。

**2. 三模合一**：一块板同时提供：
- **SLE**：实时控制指令（20μs）
- **WiFi6**：高清图传/大数据回传（114.7Mbps）
- **BLE**：手机调试/低功耗状态上报

**3. 国密安全**：SM2/SM3/SM4 硬件加密，适合需要数据安全的场景。

**4. 国产自主**：星闪是中国自主短距无线标准，华为/荣耀手机+问界汽车已搭载，2026年进入工业标准。

#### 与现有硬件的集成方案

**方案一：Jetson Nano + H3863 + ESP32-Cam（推荐）**

```
Jetson Nano（视觉/规划）
    ↓ UART（4Mbps）
BearPi-Pico H3863（通信中枢）
    ↓ SLE（20μs）  → 执行器/舵机/电机驱动
    ↓ WiFi6       → 上位机/手机高清图传
    ↓ BLE         → 手机调试/配网
```

接线：
- H3863 UART1 TX → Jetson Nano `/dev/ttyTHS1` RX
- H3863 UART1 RX → Jetson Nano `/dev/ttyTHS1` TX
- H3863 GND → Jetson Nano GND
- H3863 GPIO（PWM）→ 电机驱动板信号输入

**方案二：Cyber Bricks 执行层 + H3863 通信层**

Cyber Bricks 已有 ESP32-C3（WiFi+BLE），H3863 作为通信扩展模块叠加：
- H3863 通过 UART 与 Cyber Bricks 主控板通信
- H3863 通过 SLE 无线接收上位机指令
- Cyber Bricks 专注执行，H3863 专注通信

**方案三：H3863 作为智能家居网关**

H3863 作为 HomeAssistant 的本地无线网关：
- WiFi6 连接路由器（以太网回传）
- SLE 连接室内传感器节点（门锁/灯/温湿度）
- MQTT 协议与 OpenClaw Gateway 通信
- 支持国密，隐私数据不出家门

#### 自由度（Degrees of Freedom, DOF）说明

- 中指/无名指/小指：各 4 DOF
- 腕部：3 DOF
- 总计：约 27-28 DOF

## 2.8 技术可行性：五条技术路线

> 最后更新：2026-03-24

> 这件事在 2024-2026 年变得真正可行，是因为五条技术路线同时成熟，每条均为独立技术栈，对应行业关键拐点。各技术的详细实现方案详见 §5（AI 与感知）和 §6（本地 LLM 推理）。

### 技术路线一：开源 AI Agent 框架

OpenClaw 是全球最流行的开源 AI Agent 项目，被称为"个人 AI 的操作系统"。2026 年 GTC 上黄仁勋宣布 *"AI agents era has arrived"*，NVIDIA 官方推出 NemoClaw（集成 OpenShell 安全沙箱 + Nemotron 模型），AMD 推出 RyzenClaw / RadeonClaw 部署指南（128GB 统一内存跑 70B 模型），Intel 优化 OpenClaw 混合 AI 执行（本地 + 云端），腾讯推出 QClaw 深度集成微信/QQ 生态。Claw 已成为 AI Agent 领域的命名共识，各大厂商纷纷围绕 OpenClaw 构建自己的配套生态。0-1 基于 OpenClaw 构建，获得了整个行业生态的背书。

### 技术路线二：消费级物理仿真

Genesis 是华人团队（CMU、MIT 等）开源的物理仿真引擎，RTX 4090 上 43M FPS（RTX 5090 已发布，性能更强，待更新 benchmark）；GTX 1080 6GB 起就能跑（Ubuntu 台式机 RTX 2060 6GB 可用），MIT 开源免费，让数字孪生训练的门槛从 Isaac Gym 的 RTX 3090+ 降到几千元。

### 技术路线三：个人制造工具链

拓竹 H2C + Bambu Suite + Cyber Bricks，让一个人在家里就能完成从 3D 打印结构件到精密加工到运动控制的全流程。

### 技术路线四：本地大模型推理

Ollama + GGUF (Q4_K_M) 让 7B 模型在现有台式机 RTX 2060 上跑；vLLM / LM Studio / SGLang 各有优势——Ollama 最易用、vLLM 吞吐最高、SGLang 支持复杂 Agent 任务；AMD AI Halo 128GB（618 可能低于 1 万）跑 Qwen3.5-122B。所有推理完全本地，贵庚的记忆系统可以完全本地化，隐私安全。

### 技术路线五：统一内存架构芯片走向成熟

苹果 M 系列、AMD AI Max 系列都采用 CPU/GPU/NPU 共享物理内存的统一内存架构，避免了传统 CPU 与独立显存之间的数据搬运开销——苹果 M5 带宽 153GB/s、MacBook 24GB 可跑 30B 模型（4-bit 量化），AMD AI Max+ 128GB 统一内存、256GB/s 带宽、可跑 70B 参数模型；英特尔 Lunar Lake 也尝试了封装内存设计，虽然已确认是短期尝试但说明行业方向一致。LPDDR5X / LPDDR6 等低电压、大容量、高带宽内存颗粒解决了传输瓶颈；PCIe 5.0 / 6.0 的高带宽打通了芯片与内存之间的最后一公里。这些技术同时成熟，让大模型本地化推理真正具备可行性。

## 3.3 通信协议

### 设备内通信

| 协议 | 用途 | 延迟 | 说明 |
|------|------|------|------|
| **MQTT** | 设备间命令传递 | <50ms | 主协议，QoS 1 保证送达 |
| **WebSocket** | Gateway ↔ 节点 | <20ms | 实时双向，OpenClaw Node 协议 |
| **REST API** | 控制指令 | <100ms | 简单场景，ESP32 有现成实现 |

### 有线通信（GPIO / UART / I2C）

| 协议 | 用途 | Jetson Nano | ESP32-Cam |
|------|------|------------|-----------|
| **UART** | 简单命令传递 | Pin 8(TX)/10(RX)，/dev/ttyTHS1 | GPIO1/3，波特率115200 |
| **I2C** | 多设备总线 | Pin 3(SDA)/5(SCL)，速率10-400kHz | GPIO4/5，需要电平3.3V |
| **GPIO** | 应急停止信号 | Pin 29 等，/sys/class/gpio | 任意GPIO，硬件中断μs级 |

> **关键洞察**：Jetson Nano 和 ESP32-Cam 都有 40 针 GPIO，这是有线控制的核心。有线应急停止 <1ms，无线 WiFi >100ms。

### 通信场景选择

| 场景 | 推荐协议 |
|------|---------|
| 紧急停止 | **有线 GPIO**（<1ms，故障安全）|
| 启动/停止时序 | **有线 UART**（115200波特率）|
| 多设备同步时钟 | **有线 I2C**（TCA9548A多路复用）|
| 传感器数据采集 | 无线 MQTT（Qos 1）|

### 3.4.2 ESP32-CAM — micro-ROS 接入方案

#### 核心限制
- **520KB SRAM / 4MB Flash**：无法跑完整 ROS 2 栈
- 标准 ROS 2 依赖 POSIX 线程、动态内存，FreeRTOS 不满足
- OV2640 JPEG 压缩可发布，但高分辨率/高帧率会占满内存

#### 方案 1：micro-ROS + WiFi（推荐）

```
ESP32-CAM (micro-ROS 节点)
    ↓ WiFi UDP
micro-ros-agent (运行在 Jetson Nano / PC)
    ↓
ROS 2 主节点
```

```cpp
// ESP32 端示例（Arduino + micro_ROS）
#include <micro_ros_arduino.h>
#include <sensor_msgs/msg/compressed_image.h>

rcl_node_t node;
rcl_publisher_t pub;
sensor_msgs__msg__CompressedImage msg;

void setup() {
  // WiFi 连接
  set_microros_wifi_transports("SSID", "password", "agent-ip", 8888);
  
  // 创建节点
  rclc_node_init_default(&node, "esp32_cam", "", &support);
  
  // 创建发布者
  rclc_publisher_init_default(&pub, &node, ROSIDL_GET_MSG_TYPE_SUPPORT(
    sensor_msgs, msg, CompressedImage), "/camera/image/compressed");
}

void loop() {
  // 采集 OV2640 JPEG → 发布
  camera_fb_t *fb = esp_camera_fb_get();
  if (fb) {
    msg.format.data = (char*)"jpeg";
    msg.data.data = fb->buf;
    msg.data.size = fb->len;
    rcl_publish(&pub, &msg, NULL);
    esp_camera_fb_return(fb);
  }
  delay(100); // 10 FPS
}
```


### 3.4.3 BearPi Pico H3863 — 移植方案

#### 现状
- Hi3863（RISC-V 32bit，240MHz，4MB Flash）**无 MMU，无法跑标准 Linux**
- **无官方 micro-ROS 包**，需基于 FreeRTOS + RISC-V 工具链自行移植
- 星闪 SLE / WiFi 6 / BLE 三模通信是 H3863 的独特优势

#### 移植路径

```
1. 搭建 Hi3863 开发环境（RISC-V 工具链 + 小熊派 SDK）
2. 移植 micro-ROS 核心（rcl / rclc，基于 FreeRTOS）
3. 适配 Micro XRCE-DDS 到 Hi3863 的 UART / SPI / WiFi 驱动
4. 主机运行 micro-ros-agent，桥接到 ROS 2
5. 星闪 SLE 作为传输层（<5ms 延迟，适合实时控制）
```

#### 星闪 SLE vs 其他无线方案对比

| 方案 | 延迟 | 可靠性 | 适用场景 |
|------|------|--------|---------|
| **星闪 SLE** | <5ms | 极高 | 机器人实时控制（推荐） |
| UWB | <1ms | 极高 | 高精度定位 + 控制 |
| BLE 5.3 | 10-20ms | 高 | 低速关节、传感器 |
| WiFi 6 | 10-30ms | 中 | 高清图传、非实时 |

```
    ↓
```

#### Swift + rosbridge 示例

```swift
import ARKit
import Starscream  // WebSocket 库

class ROS2Bridge: WebSocketDelegate {
    let socket: WebSocket
    
    init(agentHost: String, port: Int) {
        let url = URL(string: "ws://\(agentHost):\(port)")!
        socket = WebSocket(request: URLRequest(url: url))
        socket.delegate = self
        socket.connect()
    }
    
    // ARKit 每帧回调
    func session(_ session: ARSession, didUpdate frame: ARFrame) {
        // 发布 Odometry（VIO 位姿）
        let odom: [String: Any] = [
            "op": "publish",
            "topic": "/iphone/odom",
            "msg": [
                "header": ["stamp": ["sec": Int64(Date().timeIntervalSince1970)]],
                "pose": ["pose": ["position": ["x": frame.camera.transform.columns.3.x,
                                               "y": frame.camera.transform.columns.3.y,
                                               "z": frame.camera.transform.columns.3.z]]
            ]
        ]
        if let data = try? JSONSerialization.data(withJSONObject: odom) {
            socket.write(data: data)
        }
    }
    
    func didReceive(event: WebSocketEvent, client: WebSocketClient) {
        // 接收 ROS 2 发来的控制指令
        if case .text(let str) = event {
            if let data = str.data(using: .utf8),
               let json = try? JSONSerialization.jsonObject(with: data) as? [String: Any],
               json["op"] as? String == "publish" {
                // 解析指令 → 执行动作
            }
        }
    }
}
```

```python
import subprocess
import edge_tts
import asyncio
import queue
import sys

# sys.path 需要加入 openclaw-bridge 路径
sys.path.insert(0, '/home/nvidia/openclaw-bridge')

    await communicate.save("/tmp/tts_output.mp3")
    subprocess.run(["aplay", "/tmp/tts_output.mp3"])
```

## Phase 3：面部表情系统

### 设计理念
0-1 的面部由「0」「-」「1」三个核心元素构成，动态显示 AI 状态。

### 系统组成

| 组件 | 形态 | 功能 |
|------|------|------|
| 「0」| LED 点阵/IPS 屏幕 | 显示主眼睛，动态表情 |
| 「-」| 线性灯光带（NeoPixel）| 情绪光效（颜色+呼吸节奏）|
| 「1」| 小型显示屏/LED | 辅助信息、状态指示 |

### 拓竹加工方案

| 零件 | 拓竹工具 | 说明 |
|------|---------|------|
| 外壳 | H2C + PLA | 面部支架，3D 打印 |
| 眼睛屏幕支架 | H2C + PETG | 耐温 |
| 透光板 | Bambu Suite 激光切割 | 亚克力 3mm |
| 安装螺丝 | 拓竹配套 M2/M3 螺丝 | 标准件 |

### 控制代码

```python
#!/usr/bin/env python3
"""0-1 面部表情控制 - Phase 3"""
from neopixel import NeoPixel
from machine import Pin
import time

N = 12  # NeoPixel 灯珠数量
pin = Pin(15, Pin.OUT)
strip = NeoPixel(pin, N)

# 表情定义（RGB 元组）
EXPRESSIONS = {
    "idle":     [(0, 50, 100)],     # 淡蓝呼吸
    "thinking": [(50, 50, 0)],       # 淡黄
    "happy":   [(0, 100, 50)],      # 绿色
    "sad":     [(30, 0, 80)],       # 淡紫
    "alert":   [(100, 0, 0)],       # 红色闪烁
}

def show_expression(name, duration=3.0):
    colors = EXPRESSIONS.get(name, EXPRESSIONS["idle"])
    for _ in range(int(duration * 10)):
        for i in range(N):
            color = colors[i % len(colors)]
            strip[i] = color
        strip.write()
        time.sleep(0.1)

# MQTT 接收表情指令
import paho.mqtt.client as mqtt

def on_message(client, userdata, msg):
    expr = msg.payload.decode()
    show_expression(expr)

client = mqtt.Client()
client.on_message = on_message
client.connect("192.168.x.x", 1883, 60)
client.subscribe("0-1/expression")
client.loop_start()
```

```python
import network
import paho.mqtt.client as mqtt
from machine import Pin, PWM, UART
import ujson

# 初始化 UART（连接 Jetson Nano）
uart = UART(1, 115200, tx=Pin(4), rx=Pin(5))

# 初始化舵机
servo1 = PWM(Pin(15))
servo1.freq(50)

# 初始化电机驱动
motor_in1 = Pin(12, Pin.OUT)
motor_in2 = Pin(13, Pin.OUT)

def set_servo(angle):
    """角度转舵机 PWM 占空比"""
    duty = int(40 + angle * 95 / 90)  # 0-90度
    servo1.duty(duty)

def set_motor(speed):
    """速度 -100 到 100"""
    if speed > 0:
        motor_in1.value(1)
        motor_in2.value(0)
    elif speed < 0:
        motor_in1.value(0)
        motor_in2.value(1)
    else:
        motor_in1.value(0)
        motor_in2.value(0)

def on_message(topic, msg):
    try:
        cmd = ujson.loads(msg)
        if cmd['type'] == 'servo':
            set_servo(cmd['angle'])
        elif cmd['type'] == 'motor':
            set_motor(cmd['speed'])
        elif cmd['type'] == 'stop':
            set_motor(0)
            set_servo(90)
    except Exception as e:
        print(f"Error: {e}")

def mqtt_on_message(client, userdata, msg):
    on_message(msg.topic, msg.payload)

# 连接 WiFi
sta = network.WLAN(network.STA_IF)
sta.active(True)
sta.connect('SSID', 'PASSWORD')
while not sta.isconnected():
    pass

# 连接 MQTT
mqtt_client = mqtt.Client()
mqtt_client.on_message = mqtt_on_message
mqtt_client.connect("192.168.x.x", 1883)
mqtt_client.subscribe("0-1/cyberbrick1")
mqtt_client.loop_start()
```

### OpenClaw → Cyber Bricks 指令脚本

```python
#!/usr/bin/env python3
"""OpenClaw 指令到 Cyber Bricks - Phase 4"""
import paho.mqtt.client as mqtt
import sys

MQTT_BROKER = "192.168.x.x"  # Jetson Nano
TOPIC = "0-1/cyberbrick1"

def send_command(action, **kwargs):
    payload = {"action": action, **kwargs}
    client = mqtt.Client()
    client.connect(MQTT_BROKER, 1883)
    client.publish(TOPIC, str(payload))
    client.disconnect()

if __name__ == "__main__":
    action = sys.argv[1] if len(sys.argv) > 1 else "stop"
    if action == "forward":
        send_command("motor", speed=50)
    elif action == "backward":
        send_command("motor", speed=-50)
    elif action == "left":
        send_command("servo", angle=45)
    elif action == "right":
        send_command("servo", angle=135)
    elif action == "stop":
        send_command("stop")
```

## 5.1 Jetson Nano 视觉感知

> 最后更新：2026-03-21

> ⚠️ **Jetson Nano 2GB 版本特别说明**：与 4GB 不同，2GB 必须开启 swap 才能同时跑 YOLO + MediaPipe，且 GPU 加速仅支持 FP16（Maxwell 不支持 INT8）。

> 📌 **手部与执行器设计** 详见 §2.7。

### 工具选择

| 工具 | 功能 | 适用场景 | Nano 2GB 适配 |

### YOLO + TensorRT FP16 部署（Nano 2GB 必读）

### 实测 FPS（完整 pipeline，含前后处理）

| 模型 | 输入分辨率 | FPS | 说明 |
|------|----------|-----|------|
| YOLOv8n | 640x640 | ~13-16 | 推理~19FPS |
| YOLOv8n | 416x416 | ~20-25 | 降分辨率换速度 |
| YOLOv8n | 320x320 | ~35-45 | 极端优化 |

> ⚠️ **INT4 在 Nano 2GB 上不可用**（Maxwell 架构不支持 INT8 推理加速）。**FP16 是唯一有效加速**。

### MediaPipe on Jetson Nano 2GB

**结论：GPU 加速方案不成熟，不推荐在 Nano 上跑 MediaPipe GPU。**

- Jetson Orin Nano 上 MediaPipe 手势也只有 ~5 FPS
- 替代方案：用 **TensorRT** 替代（ncnn + Vulkan 后端可用）
- 推荐：Pose/手势用 MobileNet + TensorRT FP16 替代 MediaPipe

### YOLO + MediaPipe 共存方案

| 配置 | FPS | 内存占用 |
|------|-----|---------|
| MediaPipe Pose（GPU）| ~20 FPS | ~500-800 MB |
| YOLOv5n + TensorRT FP16 | ~15-20 FPS | ~1-1.5 GB |
| **两者同时跑** | **约 10-15 FPS** | **~2.5 GB（需开swap）** |

**推荐串联架构**：
```
输入帧 → YOLOv5n 检测人体区域 → MediaPipe Pose 姿态估计
```

#### 3D Gaussian Splatting（3DGS）：空间智能的重建底座 ⭐⭐⭐⭐⭐

> **补充时间**：2026-03-24 | **来源**：arXiv + GitHub + 多源搜索验证
>
> World Labs 的 3D 世界生成底层依赖 3DGS（3D Gaussian Splatting）技术。3DGS 自 SIGGRAPH 2023 提出后，在机器人领域快速拓展，**对 0-1 的价值超出预期**。

**核心原理**：用数百万个 3D 高斯椭球表示空间，每个高斯含位置/形状/颜色/不透明度，通过可微分光栅化渲染，比 NeRF 快 100 倍以上（1080p @ 100+ FPS）。

##### 核心优势对比

| 指标 | NeRF | **3DGS** |
|------|------|---------|
| 训练时间 | 数小时~数天 | **35-45 分钟** |
| 1080p 渲染 | ~10 FPS | **100+ FPS** |
| 场景表示 | 隐式（神经网络）| **显式（高斯点云）** |
| 编辑/控制 | 困难 | **容易**（可编辑单个高斯）|

##### 主要应用方向

**① 实时高保真 3D 建图（LangSplat）** ⭐⭐⭐⭐⭐

GitHub：https://github.com/minghanqin/LangSplat（CVPR 2024 Highlight）

将 CLIP 视觉语言特征注入 3DGS，实现**开放词汇语义查询**：
```
用户语音："去沙发旁边那个小边几上拿我的药"
↓
LangSplat 自然语言查询："沙发" + "边几" + "药" → 3D 坐标
↓
0-1 导航到该位置 → 抓取
```
优势：不需要预定义物体类别，直接用自然语言搜索任意物体。

**② Real2Sim 仿真训练（RoboGSim）** ⭐⭐⭐⭐⭐

论文：arXiv:2411.11839，https://robogsim.github.io/

Pipeline：
```
① 手机拍摄机器人工作区 → 3DGS 高保真重建
② 数字孪生构建（Scene Composer 编辑场景）
③ 在仿真中批量生成合成数据（1000+ 条）
④ 训练机械臂策略 → 零样本迁移到真实机器人
```
关键结果：在仿真数据上训练的模型，新场景下**优于真实数据训练的模型**，解决真实数据采集成本高、场景单一的问题。

**③ 物理仿真 + 精细操作（Physical AI）**

3DGS 高斯可绑定物理引擎参数（摩擦力、形变阈值），实现：
- 机械臂抓取路径仿真（精度 0.1mm）
- 软体操作（收拾衣物、整理桌面）
- 仿真中测试 100 种抓取路径 → 选择最优下发执行

## 5.5 机器人技能训练：从模仿学习到自主泛化

> 最后更新：2026-03-24

> ⚠️ **0-1 当前状态**：文档完全没有描述"如何让机器人学会新任务"。这是具身智能的核心能力，必须补充。

#### 行业现状

| 厂商 | 训练方式 | 特点 |
|------|---------|------|
| Tesla | **模仿学习 + 端到端 FSD** | 看人类演示15分钟即可学会新任务，复用 FSD 神经网络 |
| Boston Dynamics | **Orbit 平台一键部署** | 单机学会新技能后，一键同步到整个 fleet |
| Figure AI | **端到端神经网络** | 完全自主，不依赖预编程 |
| 智元机器人 | **GO-1 通用具身基座模型** | 跨场景多任务执行，开源 |
| 宇树 | **强化学习（RL）** | 强化学习驱动程序，后空翻、侧空翻等高难度动作 |

#### 三种主要训练范式

**1. 模仿学习（Imitation Learning）**——最适合 0-1 阶段一

原理：人类通过 VR 手柄/外骨骼/视觉示范动作，机器人观察学习。

```
人类示范（VR手柄/动作捕捉）→ 机器人观察 → 神经网络学习映射 → 机器人复制动作
```

优点：样本效率高（15-30分钟学会新任务），不需要从头强化学习
缺点：泛化能力有限，新场景需要额外数据

工具：
- **Rematrix VR 手柄**（参考方案）：¥200-500，通过 VR 手柄记录6DOF姿态
- **Kinect / iPhone LiDAR**：捕捉人体姿态，成本极低
- **遥操作演示**：直接用键盘/手柄控制机器人，记录动作序列

**2. 强化学习（Reinforcement Learning）**——适合运动技能

原理：在仿真环境（isaac sim / PyBullet）中训练，迁移到真实机器人。

```
仿真环境（数字孪生）→ 强化学习训练 → sim-to-real 迁移 → 真实机器人部署
```

优点：可探索极端动作（空翻、跌倒恢复），安全无风险
缺点：sim-to-real 差距大，需要大量工程调参
适用场景：运动技能（行走、跑步、翻身）

工具：
- **NVIDIA Isaac Gym**（参考）：GPU 加速 RL 训练，宇树等厂商使用
- **PyBullet**（开源）：入门级仿真环境
- **MJX（MuJoCo）**：学术界标准，刚被 DeepMind 开源

**3. 端到端神经网络（End-to-End）**——Tesla/Figure 路线

原理：直接从传感器（摄像头/触觉）到动作输出，不用中间表示。

```
视觉输入 → 端到端 NN → 动作输出（关节扭矩/位置）
```

优点：无需手工设计特征，泛化能力强
缺点：需要海量数据（Tesla 用 FSD 数据飞轮），算力要求极高
适用场景：通用家庭机器人（Figure 03、Tesla Optimus）

**4. 3DGS 仿真数据生成（RoboGSim）**——解决数据不足的关键方案

原理：用 3D Gaussian Splatting 重建真实场景，在仿真中批量生成合成动作数据。

```
① 手机拍摄任务场景 → 3DGS 高保真数字孪生
② 在仿真中添加/删除/移动物体，修改光照
③ 批量生成 1000+ 条合成动作数据
④ 在仿真中训练机器人策略
⑤ sim-to-real 迁移到真实 Cyber Bricks
```

优点：
- 数据量可无限扩展（不受真实采集成本限制）
- 可控制场景变量（光照/物体材质/遮挡程度）
- 在仿真中训练的模型，新场景下表现**优于真实数据训练的模型**
- 解决 0-1 最大痛点：Cyber Bricks 真实数据采集成本高

缺点：
- 需要 RTX 4090 最佳（RTX 5090已发布，待更新；RTX 2060 凑合但慢）
- sim-to-real 纹理迁移仍有少量差距

关键项目：
- **RoboGSim**（https://robogsim.github.io/）：完整 Real2Sim2Real Pipeline，arXiv:2411.11839

#### 0-1 分阶段训练方案

**阶段一（模仿学习为主）**

目标：让机器人学会日常任务（递东西、倒水、收拾桌面）

工具链：
1. 用键盘/手柄遥操作记录动作序列（成本 ¥0）
2. 用 Python 脚本将动作序列转为标准化格式
3. 用小模型（如 Qwen2.5-7B）做动作理解 + 规划
4. 通过 MQTT 下发动作指令到 Cyber Bricks

关键限制：Cyber Bricks 只有基础舵机，无触觉传感，精细操作受限

**阶段二（强化学习补充）**

目标：学会运动技能（行走恢复、跌倒站起、上下楼梯）

工具链：
1. 在 PyBullet/MJX 中建立 Cyber Bricks 仿真模型
2. 用 PPO 算法训练运动策略
3. Sim-to-real 迁移（可能需要 domain randomization）
4. 部署到真实 Cyber Bricks

**阶段三（端到端 AI）**

目标：通用家庭服务能力（类 Figure 03 / Optimus）

工具链：
1. 收集家庭场景真实数据（视觉 + 动作配对）
2. 用多模态大模型（如 LLaVA/Qwen-VL）做视觉-语言-动作对齐
3. 部署到 AGX Thor 或 AI Halo（算力要求高）

#### 关键问题：数据从哪来？

| 数据来源 | 方式 | 成本 | 质量 |
|---------|------|------|------|
| **遥操作演示** | 人工控制，记录动作 | ¥0 | 高（人工筛选）|
| **视频数据** | YouTube/互联网大规模视频 | 低 | 中（需要过滤）|
| **仿真生成** | NVIDIA Isaac Gym / Genesis | 高（GPU 算力）| 中（sim-to-real 差距）|

0-1 推荐：**遥操作 + RoboGSim 3DGS + Cosmos（远期）+ 15TB 数据集预训练 + 阶段三积累真实数据**

## 8.1 声纹识别

### 目标
通过声纹识别确认用户身份，只有授权用户的声音才能触发关键操作。

### 实现方式

```python
import librosa
import numpy as np

def extract_voice_embedding(audio_path):
    y, sr = librosa.load(audio_path, sr=16000)
    mfccs = librosa.feature.mfcc(y=y, sr=sr, n_mfcc=40)
    return np.mean(mfccs.T, axis=0)

def verify_speaker(test_audio, enrolled_audio, threshold=0.85):
    test_emb = extract_voice_embedding(test_audio)
    enrolled_emb = extract_voice_embedding(enrolled_audio)
    similarity = np.dot(test_emb, enrolled_emb) / (
        np.linalg.norm(test_emb) * np.linalg.norm(enrolled_emb)
    )
    return similarity > threshold
```

### 声纹注册流程

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 用户说3-5个不同句子 | 收集多样本 |
| 2 | 提取每句的声纹向量 | 10-15秒音频 |
| 3 | 平均后存储 | 存入本地数据库 |
| 4 | 验证时比对 | 余弦相似度 >0.85 通过 |

| CPU 使用率 | < 90% | > 95% 持续5分钟 | 重启进程 |
| 内存使用 | < 80% | > 90% | 清理缓存 |
| 网络延迟 | < 100ms | > 500ms | 切换网络 |
| ESP32-Cam FPS | > 15 FPS | < 10 FPS | 重启固件 |

### 监控脚本

```python
#!/usr/bin/env python3
import psutil, subprocess, time, requests

def get_cpu_temp():
    try:
        with open('/sys/class/thermal/thermal_zone0/temp') as f:
            return int(f.read()) / 1000
    except: return 0

def check_system():
    warnings = []
    temp = get_cpu_temp()
    if temp > 85: warnings.append(f"CPU温度过高: {temp}C")
    mem = psutil.virtual_memory()
    if mem.percent > 90: warnings.append(f"内存不足: {mem.percent}%")
    cpu = psutil.cpu_percent(interval=5)
    if cpu > 95: warnings.append(f"CPU过载: {cpu}%")
    return warnings

def send_alert(message):
    try:
        requests.post("http://192.168.1.x:3000/api/alert",
            json={"type": "system_warning", "message": message}, timeout=5)
    except: pass

while True:
    warnings = check_system()
    for w in warnings:
        print(f"WARNING: {w}")
        send_alert(w)
    time.sleep(60)
```

### 目标
在物理被盗或被强制夺取时，自动销毁贵庚的核心数据。

### 触发条件

| 触发方式 | 说明 |
|---------|------|
| 物理按钮 | 机器人被夺走时按下 |
| 遥控指令 | 发送销毁信号 |

| 超时未认证 | 超过设定时间未声纹验证（详见 §1.2：默认30天，可设定） |

### 销毁流程

```python
#!/usr/bin/env python3
def destroy_all_data():
    import os, subprocess
    DATA_DIRS = [
        "/home/nvidia/guigeng/memory",
        "/home/nvidia/guigeng/videos",
        "/home/nvidia/guigeng/audio",
    ]
    print("WARNING: 触发数据自毁程序")
    subprocess.run(["pkill", "-f", "robot-voice"])
    subprocess.run(["pkill", "-f", "robot-video"])
    for dir_path in DATA_DIRS:
        if os.path.exists(dir_path):
            subprocess.run(["shred", "-n", "3", "-ruv", dir_path])
    print("OK: 数据自毁完成")
```

> WARNING: 数据自毁后不可恢复。重要数据应定期备份到 NAS。
> ⚠️ **SSD/NVMe 限制**：`shred -n 3` 对 SSD/NVMe 效果有限（ Wear Leveling 会分散数据），推荐方案：全程使用 **full disk encryption（LUKS）**（部署时在 §2 硬件体系中统一配置），失控时直接销毁密钥而非数据本身。
