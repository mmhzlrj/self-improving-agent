## 关键发现

### 1. 自我验证的盲区
MiniMax 用自己的知识验证自己的审核 → 对"新信息"容易漏判。
- V01: MiniMax 的知识截止日期不包含 2026 年 2 月 Qwen3.5 发布
- DeepSeek 有联网搜索 → 查到阿里云官方文档 → 判定正确

**结论：事实性验证必须用联网搜索能力，不能依赖模型自身的知识库。**

### 2. 误报模式（3 种）
1. **知识过时**: V01 — MiniMax 知识截止日期晚于 Qwen3.5 发布
2. **反向误判**: V04 — MiniMax 说文档错了，实际文档是对的
3. **过度质疑**: V12 — MiniMax 怀疑正确地址，找不到就说是错的

### 3. 五平台验证策略有效
- DeepSeek: 有深度思考和联网搜索，发现 V01
- 豆包: 精确技术规格验证（Hi3863V100、ROS2）
- Kimi: 学术性事实（DOF、Whisper）验证
- GLM: 产品信息（Jetson Thor、Phi-3.5）验证
- 千问: 超时未返回（可作为降级策略）

## 推荐的两步法工作流

### 第一步：MiniMax Subagent 审核
```
20 个 subagent 并发 → exec+sed 读取 → 逐段审核 → 输出 result.json
```

### 第二步：五平台交叉验证（GLM 主 agent 直接调用）
```
筛选事实性声明 → 4-5 个聊天工具并行验证 → 交叉对比 → 输出最终裁决
```

### 为什么不用 MiniMax subagent 做第二步？
1. MiniMax 无联网搜索能力（web_search 质量有限）
2. 自我验证有盲区（用自己的知识验证自己的判断）
3. 五平台（DeepSeek/Kimi/豆包/GLM/千问）各有联网搜索，交叉验证更可靠

### 额度消耗
- 第一步: ~800k MiniMax tokens（20 个 subagent）
- 第二步: ~5 个聊天工具调用（消耗各平台自己的 tokens，不消耗 GLM tokens）
- **GLM tokens 消耗: 极低**（只负责调度和汇总）
