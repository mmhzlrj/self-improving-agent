## 一、概述

本报告定义 iPhone 感知前端 → Robot Controller 的数据传输格式，覆盖：

| 数据流 | 来源 | 目标帧率 | 单帧原始大小（估算） |
|--------|------|----------|----------------------|
| 深度图（Depth Map） | LiDAR Scene Depth API | 60 Hz | ~11 MB（1920×1440 Float32） |
| RGB 图像（Camera Frame） | ARKit ARFrame | 30-60 Hz | ~6.2 MB（1920×1440 JPEG@80%） |
| IMU 数据（Accelerometer + Gyroscope） | CoreMotion | 100 Hz | ~2.4 KB/帧 |
| 设备姿态（Camera Transform） | ARKit World Tracking | 60 Hz | 256 B（16×Float32） |
| 人体骨骼（Body Anchor） | ARKit Body Tracking | 30 Hz | ~1.4 KB |
| 面部表情（Face Anchor） | ARKit Face Tracking | 60 Hz | ~600 B |

## 三、深度图（Depth Map）

### 3.1 ARKit 原生格式

```swift
// ARKit Scene Depth — 每帧获取方式
if let depth = frame.sceneDepth {
    let depthMap    = depth.depthMap     // CVPixelBuffer, Float32, 1 channel, 单位：米
    let confidence  = depth.confidenceMap // CVPixelBuffer, UInt8, 1 channel, 0=低/1=中/2=高
    let intrinsics   = depth.cameraIntrinsics // simd_float3x3: [fx,0,cx; 0,fy,cy; 0,0,1]
}
```

### 3.2 传输格式

| 字段 | 类型 | 说明 |
|------|------|------|
| `timestamp` | `uint64` | CVTimestamp（Nanoseconds since boot） |
| `frameId` | `uint32` | 递增帧序号 |
| `width` | `uint16` | 深度图宽度（像素） |
| `height` | `uint16` | 深度图高度（像素） |
| `data` | `bytes` | Float32 数组，行主序，z=0 表示无效 |
| `confidence` | `bytes` | UInt8 数组，对应每个像素的置信度 |
| `intrinsics` | `float[9]` | 相机内参矩阵（按行展开） |
| `extrinsics` | `float[16]` | 相机外参（simd_float4x4，按列展开） |

**原始大小计算（1920×1440）**：
- `data`: 1920 × 1440 × 4 bytes = **~11 MB**
- `confidence`: 1920 × 1440 × 1 byte = **~2.8 MB**

### 3.3 压缩方案

| 方案 | 压缩比 | 解压延迟 | 适用场景 |
|------|--------|----------|----------|
| **zlib（默认）** | ~3-5× | < 5 ms | 实时流（本地 Socket） |
| **zstd** | ~4-6× | < 8 ms | 局域网 WiFi |
| **LCoS（Squeeze）** | ~10× | < 10 ms | 低带宽（4G） |
| **JPEG-D（ depth JPEG）** | ~8-10× | < 15 ms | 开发/Debug |

**推荐**：实时流用 **zlib** level=3（速度与压缩比平衡），压缩后 ~2-4 MB/帧。

### 3.4 深度图特殊处理

- **z=0 像素**（无效深度）：ARkit 深度图有效范围 0.05m–5m，z=0 表示无数据
- **置信度过滤**：仅传递 confidence ≥ 1（中/高）的像素，低置信度区域填充相邻均值
- **空间下采样**：若带宽紧张，可 2×2 合并为 960×720，节省 75% 带宽（仍足够避障精度）

## 四、RGB 图像

### 4.1 ARKit 原生格式

```swift
// ARFrame → CPU 端图像
let pixelBuffer = frame.capturedImage  // CVPixelBuffer, CVPixelFormatType: kCVPixelFormatType_32BGRA

// 转换为 JPEG（压缩）
let ciImage = CIImage(cvPixelBuffer: pixelBuffer)
let jpegData = JPEGEncoder.encode(ciImage, quality: 0.8)  // ~500 KB
```

### 4.2 传输格式

| 字段 | 类型 | 说明 |
|------|------|------|
| `timestamp` | `uint64` | 与深度图同源（CVTimestamp） |
| `frameId` | `uint32` | 递增帧序号（与深度图同步） |
| `width` | `uint16` | 图像宽度 |
| `height` | `uint16` | 图像高度 |
| `format` | `uint8` | 0=JPEG, 1=PNG, 2=CVPixelBuffer 直接序列化 |
| `quality` | `uint8` | JPEG 质量 0-100 |
| `data` | `bytes` | 压缩图像数据 |
| ` intrinsics` | `float[9]` | 相机内参（同深度图） |

**JPEG 建议参数**：
- `quality=0.8`（80%）→ ~500-800 KB/帧（1920×1440）
- `quality=0.6`（60%）→ ~300-500 KB/帧，带宽紧张时

### 4.3 深度图 + RGB 同步

ARKit 的 `ARFrame` 天然保证深度图与 RGB 图像**时间同步**（同一帧）：
- 接收端通过 `frameId` 匹配
- 两者 `timestamp` 误差 < 1 ms（由 ARKit 硬件同步保证）

## 五、IMU 数据（CoreMotion）

### 5.1 ARKit IMU 数据来源

ARKit 内部已融合 IMU + 视觉里程计，但若需要**原始 IMU 数据**：

```swift
import CoreMotion

let manager = CMMotionManager()
manager.deviceMotionUpdateInterval = 1.0 / 100  // 100 Hz

manager.startDeviceMotionUpdates(to: .main) { motion, error in
    guard let m = motion else { return }

    // 加速度（世界坐标系，去除重力）
    let acc = m.userAcceleration  // simd_float3 (m/s²)

    // 旋转速率（陀螺仪）
    let gyro = m.rotationRate      // simd_float3 (rad/s)

    // 磁场方向（绝对航向参考，可选）
    let mag = m.magneticField      // simd_float3 (μT)

    // 设备姿态（四元数）
    let att = m.attitude           // CMAttitude: pitch/roll/yaw 或四元数
}
```

### 5.2 传输格式（每 10 ms 一帧，100 Hz）

| 字段 | 类型 | 单位 | 说明 |
|------|------|------|------|
| `timestamp` | `uint64` | ns | 郭端时间戳 |
| `seq` | `uint32` | - | 递增序号 |
| `acc_x/y/z` | `float` | m/s² | 加速度（去除重力） |
| `gyro_x/y/z` | `float` | rad/s | 陀螺仪角速度 |
| `mag_x/y/z` | `float` | μT | 磁场（可选） |
| `qw/qx/qy/qz` | `float` | - | 设备姿态四元数 |

**单帧大小**：6×float + 4×float + 3×float + 1×uint64 + 1×uint32 = **44 bytes**

### 5.3 批量打包（减少协议 overhead）

每 **10 帧**（100 ms）打一个 Batch 包：

```json
{
  "type": "imu_batch",
  "startTs": 1234567890123,
  "seq": 1000,
  "samples": [
    { "dt": 0,      "acc": [0.1, -0.2, 9.8], "gyro": [0.01, 0.0, 0.0], "quat": [1, 0, 0, 0] },
    { "dt": 10000,  "acc": [0.1, -0.2, 9.8], "gyro": [0.01, 0.0, 0.0], "quat": [1, 0, 0, 0] },
    // ... 10 samples
  ]
}
```

## 六、时间同步方案

### 6.1 问题

iPhone 感知前端与 Robot Controller 运行在**不同设备/时钟域**：
- iPhone：CVTimestamp（Boot Nanoseconds）
- Robot Controller：本地 system clock（Unix epoch ms 或自定义 TAI）

### 6.2 同步方案：NTP + 漂移补偿（推荐）

```
Step 1：建立连接时交换 hello 消息（含各自本地时间）
Step 2：计算时钟偏移量 offset = T_robot - T_iphone（郭端时间）
Step 3：后续所有帧的 timestamp 减去 offset 转为 Robot 时钟域
Step 4：周期性（每 10s）重新校准 offset（抵消晶体漂移）
```

**误差预算**：
- NTP 往返延迟误差：< 1 ms
- 晶体漂移：~50 ppm（10s 校准间隔 → ~0.5 ms 漂移）
- **总同步误差**：< 2 ms（满足实时控制需求）

### 6.3 包头设计（统一）

所有帧共用同一个 **FrameHeader**，通过 `msgType` 区分数据类型：

```
+---------------+---------------+---------------+----------------+
|  magic (4B)  |  version (1B) |  msgType (1B) |  flags (1B)   |
+---------------+---------------+---------------+----------------+
|              frameId (uint32)                               |
+-----------------------------------------------------------+
|              timestamp (uint64, ns, robot clock domain)    |
+-----------------------------------------------------------+
|              payloadLength (uint32)                        |
+-----------------------------------------------------------+
|              reserved (uint32)                            |
+-----------------------------------------------------------+
|                    payload ...                             |
+-----------------------------------------------------------+
```

| msgType | 数据类型 |
|---------|----------|
| 0x01 | DepthMap |
| 0x02 | RGBImage |
| 0x03 | IMU_Batch |
| 0x04 | DevicePose |
| 0x05 | BodyFrame |
| 0x06 | FaceFrame |
| 0xFE | Heartbeat |
| 0xFF | Error |

**Magic**: `0x41524B49`（"ARKI"）

## 八、Protocol Buffer 定义（可选高性能方案）

当使用 **gRPC** 或 **Protobuf** 时，替代 JSON Schema：

```protobuf
syntax = "proto3";

package iphone感知;

message FrameHeader {
  uint32 magic = 1;      // 0x41524B49
  uint32 version = 2;     // 1
  uint32 msg_type = 3;   // 见 msgType 枚举
  uint32 frame_id = 4;
  uint64 timestamp = 5;  // robot epoch ns
  uint32 payload_len = 6;
}

enum MsgType {
  DEPTH_MAP = 0x01;
  RGB_IMAGE = 0x02;
  IMU_BATCH = 0x03;
  DEVICE_POSE = 0x04;
  BODY_FRAME = 0x05;
  FACE_FRAME = 0x06;
}

message DepthMapFrame {
  uint32 frame_id = 1;
  uint64 timestamp = 2;
  uint32 width = 3;
  uint32 height = 4;
  bytes data = 5;        // Float32, row-major, z=0=invalid
  bytes confidence = 6;   // UInt8 per pixel
  float intrinsics[9] = 7;
  float extrinsics[16] = 8;
  string compression = 9; // "zlib" | "zstd" | "none"
}

message RGBImageFrame {
  uint32 frame_id = 1;
  uint64 timestamp = 2;
  uint32 width = 3;
  uint32 height = 4;
  string format = 5;     // "jpeg" | "png" | "raw"
  uint32 quality = 6;    // 1-100
  bytes data = 7;
}

message IMUSample {
  uint64 dt_ns = 1;          // delta from batch start
  float accel_mps2[3] = 2;   // [x, y, z]
  float gyro_rads[3] = 3;    // [x, y, z]
  float quat[4] = 4;         // [w, x, y, z]
}

message IMUBatchFrame {
  uint64 start_ts = 1;
  uint32 seq = 2;
  repeated IMUSample samples = 3;
}

message DevicePoseFrame {
  uint32 frame_id = 1;
  uint64 timestamp = 2;
  float transform[16] = 3;  // column-major simd_float4x4
  string tracking_state = 4;
  string world_mapping_status = 5;
}

message JointData {
  string name = 1;
  float transform[16] = 2;
  float local_pos[3] = 3;
}

message BodyFrame {
  uint32 frame_id = 1;
  uint64 timestamp = 2;
  bool is_tracked = 3;
  float root_transform[16] = 4;
  repeated JointData joints = 5;
}

message BlendShapeCoefficient {
  string name = 1;
  float value = 2;
}

message FaceFrame {
  uint32 frame_id = 1;
  uint64 timestamp = 2;
  float transform[16] = 3;
  repeated BlendShapeCoefficient blend_shapes = 4;
  float look_at_point[3] = 5;
}
```

## 九、传输协议选择

| 场景 | 推荐协议 | 说明 |
|------|----------|------|
| **本地（同一设备/局域网）** | Unix Domain Socket | 延迟 < 1 ms，带宽无限制 |
| **WiFi 局域网** | WebSocket + Binary Frame | 跨平台，支持 JSON 或 Protobuf |
| **远程/互联网** | HTTP/2 + gRPC + Protobuf | 拥塞控制，支持流式 |
| **开发/Debug** | WebSocket + JSON | 便于调试，可直接 Wireshark 看内容 |

**帧组装（批量）策略**：
- IMU 数据：每 10 帧（100 ms）打包发送，减少 UDP/TCP overhead
- 深度图/RGB：逐帧发送（延迟敏感）
- 姿态/骨骼：逐帧发送（60/30 Hz）

## 十二、实现要点

### 12.1 iOS 端（Swift）

```swift
// 双缓冲避免丢帧
let depthQueue = DispatchQueue(label: "com.robot.depth", qos: .userInteractive)
let rgbQueue   = DispatchQueue(label: "com.robot.rgb",   qos: .userInteractive)
```
